# 机器定理证明

**机器定理证明（Automated / Interactive Theorem Proving）** 或者说“使用计算机来辅助证明定理”，是指借助计算机来书写、检验乃至自动发现（并证明）数学证明的一类技术。
它的核心思想与“使用计算机来辅助计算”（各种编程语言如C/C++、Rust）相似，都是将自然语言描述的计算/证明过程转化为一种变成语言。
使得我们可以借助计算机的运算能力与准确性来高效准确的完成计算和证明。

按照人与机器的分工，大致分为两类：

- **自动定理证明（ATP, Automated Theorem Proving）**：机器自主搜索证明，典型代表是 SAT/SMT 求解器（如 Z3、CVC5）和一阶逻辑证明器（如 E、Vampire）。它们擅长有限/可判定片段，但难以处理深奥的数学。这不是本文的重点。
- **交互式定理证明（ITP, Interactive Theorem Proving）**：由人给出证明大纲、计算机负责检查每一步并补全细节，代表系统包括 **Lean**、Coq/Rocq、Isabelle/HOL、Agda 等。它能够覆盖任意复杂的现代数学。

而本文主要以[lean](https://leanprover-community.github.io/learn.html)为引子来介绍交互式定理证明，其目标式可以使用lean进行简单的证明书写。

## 基础知识与前置技能

阅读本文需要掌握一些基本的知识：

1. 能够阅读C/C++的基本语法
2. 能够使用自然/数学语言写出一些基本的证明
3. 可以使用简单的三段式逻辑推理

## 使用计算机来辅助计算

在开始使用lean证明各种东西之前，我们先看一个“使用计算机来辅助计算”的例子。 这有助于理解“使用计算机来辅助证明定理”。

### 计算$1+2+3+...+100=?$

如果我们想计算

$$
0+1+2+...+100=?
$$

我们可以在不借助任何计算工具的条件下写出的计算过程

$$
\begin{aligned}
&0+1+2+3+...+100 \\
=&(0+1)+2+3+...+100 \\
=&(1+2)+3+...+100 \\
=&(3+3)+...+100 \\
=&... \\
=&5050
\end{aligned}
$$

这个计算过程非常的直观，就是从左到右一个个数加过去。朴实而原始。那么如果我们考虑“使用计算机来辅助计算”，我们就会得到如下程序

```c++
unsigned int sum_one_to_one_hundred() {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= 100; i++) {
        sum += i;
    }
    return sum;
}
```

这段程序也非常的直观，只是将前面人肉计算的过程翻译成了计算机程序罢了。
但这完成了一个非常了不起的过程，我们第一次进行了“使用计算机来辅助计算”，计算机代替我们完成了繁琐的计算过程。

### 计算$0+1+2+...+n=?$

更进一步如果将问题改为

$$1+2+3+...+n=?$$

这时如果我尝试进行人肉计算

$$
\begin{aligned}
&0+1+2+...+n \\
=&1+2+...+n \\
=&3+...+n \\
=&... \\
\end{aligned}
$$

会发现我并不知道该加到多少停止，因为这个问题里并没有给出$n$的具体数值。因此只能说

$$请依照加法的计算方法一个个加过去，直到加到了n$$

当然如果我们“使用计算机来辅助计算”，依然可以写出

```c++
unsigned int sum_zero_to_n_directly(unsigned int n) {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= n; i++) {
        sum += i;
    }
    return sum;
}
```

“使用计算机来辅助计算”并不在乎是否给出了一个具体的$n$。因为它是上面人肉计算的一种翻译。这个程序只是将“$请依照加法的计算方法一个个加过去，直到加到了n$”这句话翻译成了计算机理解的形式。

当然经过一些思考还可以写出

```c++
unsigned int sum_zero_to_n_formula(unsigned int n) {
    unsigned int sum = n * (n + 1) / 2;
    return sum;
}
```

这个算法就有些不同了，因为它不再是人肉计算的简单翻译结果。而是使用了更高级的结论简化了计算过程。具体来说是应用了

$$
    0+1+2+...+n = \frac{n(n+1)}{2}
$$

这个结论非常的基本，以至于我们我们不会怀疑它的正确性。毕竟这已经是书本上学过并验证过的的东西了。

## 使用计算机来辅助证明

那么下面我们就以求和公式为例演示如何“使用计算机来辅助证明”。那么我们是如何进行证明的呢？一种方式是直接使用数学归法。

命题： 对任意的自然数$n$，有 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立

证：

对的自然数$n$使用数学归纳法

1. 当 $n=0$ 时有 $ 0=\frac{0\times (0+1)}{2}$ ，成立。
2. 假设$n=k$时结论成立，即 $0+1+2+...+k = \frac{k(k+1)}{2}$ 成立。下证明 $n=k+1$ 时有 $0+1+2+...+k+(k+1) = \frac{(k+1)(k+2)}{2}$ 成立。直接计算有

    $$
    \begin{aligned}
          &0+1+2+...+k+(k+1) \\
         =&(0+1+2+...+k)+(k+1)&加法结合律\\
         =&\frac{k(k+1)}{2}+(k+1)&使用归纳假设\\
         =&\frac{k(k+1)+2(k+1)}{2}&通分\\
         =&\frac{(k+2)(k+1)}{2}&提取公因式\\
         =&\frac{(k+1)(k+2)}{2}&乘法交换律\\
    \end{aligned}
    $$

    因此 $n=k+1$ 时结论成立。

    由(1)(2)与归纳法知 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立。$\square$

那么下面我们就把这个证明过程翻译为计算机理解的方式。

### 描述问题

为了方便观察与更加精确的表述问题，像上面使用自然语言去描述问题通常是不够的。因此将命题改写为如下形式。

命题：

$$
    \forall n\in \mathbb{N},\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

> 注：这个过程一般被称为问题的形式化，其目的是在于消除表达的歧义。
当然它也会带来副作用，在训练不足时会觉得这种表述非常别扭且难以理解。
这是正常的，因为形式化表述本身对于熟悉自然语言表述的人来说并不是自然而然的。
不如说任何表述或者规范本身都有其门槛和表达范式。

观察这个命题可以发现它是由命题部分和计算部分组成：

首先拆解第一个命题部分

$$
\forall A, B
$$

其中 $A$ 是 $n \in \mathbb{N}$， $B$ 是 $\sum_{i=1}^n i = \frac{n(n+1)}{2}$。
它描述的是对于任意的 $A$ 条件成立则 $B$ 结论成立。一般我们将其读作对任意的 $A$ 则 $B$ 。

再拆解 $B$ 这个部分

$$
\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

它由下面三个部分组成，

$$
\begin{aligned}
\sum_{i=1}^n i &&  \frac{n(n+1)}{2} && C = D
\end{aligned}
$$

其中$\sum_{i=1}^n i$、$\frac{n(n+1)}{2}$都是一个计算过程， 而$C=D$是一个命题。

综上，我们得到了命题部分：

$$
\begin{aligned}
 \forall A, B && C = D
\end{aligned}
$$

和计算部分

$$
\begin{aligned}
\sum_{i=1}^n i &&  \frac{n(n+1)}{2}
\end{aligned}
$$

#### 表述计算部分

下面首先使用lean表达计算部分 $\frac{n (n + 1) }{ 2}$

```{lean-playground} :height: 200px
def sum_zero_to_n_formula (n : Nat) : Nat :=
    n * (n + 1) / 2

#eval sum_zero_to_n_formula 100
```

其中```sum_zero_to_n_formula```表示了计算部分$\frac{n(n+1)}{2}$。它被定义成一个函数。lean中的一个函数由下面几个部分组成：

1. 关键字```def```表示声明一个函数（计算部分）。这是lean语法的一部分呢
2. ```(n : Nat)``` 表示 ```sum_zero_to_n_formula``` 接受一个类型为```Nat```名称叫做```n```的参数。```Nat```类型在这里表示整数。
3. ```:```与```:=```中间的```Nat```表示函数```sum_zero_to_n_formula```的返回类型为一个整数类型。
4. ```:=```后面表示的是函数的实现。返回 ```n * (n + 1) / 2``` 的计算结果。这里返回实例并不需要写 ```return``` 关键字，这是与 C/C++ 不同的地方。

我们使用```#eval```这个方法来调用这个函数。在代码右侧的消息（Message）中可以看到它的输出 ```5050``` 代表着从$0$加到$100$的结果是$5050$。

其次来表述另一个计算部分 $\sum_{i=1}^n i$

```{lean-playground} :height: 200px

def sum_zero_to_n_directly (n : Nat) : Nat :=
  if n = 0 then 
    0 
  else 
    sum_zero_to_n_directly (n - 1) + n

#eval sum_zero_to_n_directly 100
```

可以看到我们表述的是递归定义

$$
f(n) =
\begin{cases}
0, & n = 0 \\
f(n-1) + n, & n = 1,2,3,...
\end{cases}
$$

其中 $f$ 表示了函数 $sum\_zero\_to\_n\_directly$ 。
这是因为在lean中 $\sum_{i=1}^n i$ 并非不可以写成直接表达，只是递归表达相对于直接表达在证明过程中会更加容易。
换言之，lean的表达范式是推荐优先写成递归形式的。
这一点与其它编程语言刚好相反，通常在其它语言中会使用专门的技巧尽可能的消除递归表达。而在lean中请尽量保留递归表达。

#### 表述命题部分

在表述这个复杂命题之前，首先来看一些基础的命题与对应的lean表示

命题:

$$
\begin{aligned}
    1+1&=2\\
    1+1&=3\\
    \forall n\in \mathbb{N}, n &+ 1 = 4\\
    \exists n\in \mathbb{N}, n &+ 1 = 4\\
\end{aligned}
$$

```{lean-playground} :height: 300px
def prop_one_add_one_eq_two : Prop :=
  1 + 1 = 2

def prop_one_add_one_eq_three : Prop :=
  1 + 1 = 3

def prop_n_add_one_eq_four : Prop :=
  ∀ n : Nat, n + 1 = 4

def prop_exists_n_add_one_eq_four : Prop :=
  ∃ n : Nat , n + 1 = 4

```

可以看到命题的表述与之前的计算函数类似。更准确的说是有一个函数，它的返回类型为命题。
它与之前的函数最大的不同它的返回类型不再是```Nat```（自然数），而是一个新类型```Prop```(propositions，命题)。
而实例 ```1 + 1 = 2``` 、```1 + 1 = 3``` 、```∀ n : Nat, n + 1 = 4``` 和 ```∃ n : Nat , n + 1 = 4``` 都是类型```Prop``` 的实例。就如同实例 ```1``` 、```2``` 和 ```3``` 都是类型 ```Nat``` 的实例一样。

> 实例与类型
>
> 为了方便我们使用下面的记法表达实例与类型的关系。
>
> $$1 : Nat\\1 + 1 = 3 : Prop\\∃ n : Nat , n + 1 = 4 : Prop$$
>
>其中 ```:``` 左边表示实例，右边表示类型。
>分别读作
>
>$$1 是Nat的实例\\1+1=3 是Prop的实例\\存在n是Nat的实例，使得n+1=4 是Prop 的实例$$
>
>其中前两个比较简单，第三个则相对复杂，因为它涉及了嵌套关系
>
>$$n : Nat\\∃ _ , n + 1 = 4 : Prop$$
>
>将 ```∃ n : Nat , n + 1 = 4 ```这个东西视为一个整体才是```Prop```的实例。
>也就是说```存在n是Nat的实例，使得n+1=4```作为一个整体才是```Prop```的实例。
>单独拿出任何一部分都不是这个式子中期望表达的Prop的实例。
>
>>注：实际上 ```Prop``` 这个类型并没有什么特殊的意义，它和众多类型如 ```Nat``` 没有区别。只是使用lean语言的人觉得这个东西可以被证明，因此单独拿出来讨论。

对于前两个关于$1+1$的命题，显然我们知道命题： $1+1=2$ 是真命题。命题： $1+1=3$ 是假命题。
在lean中```Prop```不会限制一个假命题的表达。即命题无论真假都可以被表述。当然假命题是证不出来的。

对于后两个命题，分别表述了任意与存在。lean是数学表述的直接翻译。只是将 $\in$ 替换为了 $:$ ，其它部分并没有变化。

|lean描述|数学描述|
|-|-|
|```∀ n : Nat```|$\forall n\in \mathbb{N}$|
|```∃ n : Nat```|$\exists n\in \mathbb{N}$|

> 注：严格来说这个表述不够严谨，因为```Nat```类型并非一个集合，```n : Nat``` 表述的是 $n$ 是 ```Nat``` 的一个实例$n$。
```n ∈ (setOf fun _ : Nat => true)``` 表达的才是 $n$ 属于的由```Nat```所有元素构成的集合。
这两者是完全等价的，因此本文都选用更易于理解的一种。

接下来再来看另一组返回命题的函数

```{lean-playground} :height: 350px
def prop_nat_symm_1 (n : Nat) (m : Nat) : Prop :=
  n + m = m + n

def prop_nat_comm_2 (n m : Nat) : Prop :=
  n + m = m + n

def prop_nat_symm_3 : Prop :=
  ∀ n : Nat, ∀ m : Nat, n + m = m + n

def prop_nat_symm_4 : Prop :=
  ∀ n m : Nat, n + m = m + n

#check prop_nat_symm_1 
#check prop_nat_symm_1 1 2 
#check prop_nat_symm_4
```

其中(1)(2)是一组，命题(3)(4)是另一组。其中(1)(2)这一组，描述的是有一个函数接受两个类型为```Nat```

$$
    如果n,m是自然数，则 n+m=m+n
$$

强调的是如果你给我 $n$ 和 $m$ 则还给你命题$n+m=m+n$， 当然如果你給我 $p$ 和 $q$ 则会还给你命题 $p+q=q+p$。 
至于参数 $n$ ，$m$ 从哪里来这个命题并不关心，这一部分由构造命题的人从外部传入。

而命题(3)(4)描述的就是命题：

$$
    \forall n,m \in \mathbb{N},n+m=m+n
$$

它不依赖任何外部传入的参数，直接了当的说明想表达的命题。

在这里我们使用了 ```#check``` 来检查某个实例的类型，在lean中它与 ```#eval``` 一样会显示一些消息在message中。其中

1. ```#check prop_nat_symm_1``` 显示了 ```prop_nat_comm_1 (n m : ℕ) : Prop``` ，其含义是 ```prop_nat_comm_1``` 接受了两个参数 ```n m : ℕ``` 后返回一个的类型是```Prop```的实例。
2. ```#check prop_nat_symm_1 1 2``` 显示了 ```prop_nat_comm_1 1 2 : Prop``` ，其含义是 ```prop_nat_comm_1``` 接受了两个参数 ```1``` 与 ```2``` 后返回一个类型是```Prop```的实例。。实际上我们还知道该实例就是 ```1+2=2+1``` 。 当然```#check``` 并不显示这些。
3. ```#check prop_nat_symm_4 1 2``` 显示了 ```prop_nat_comm_4 : Prop``` ，其含义是 ```prop_nat_comm_4``` 不需要任何输入就返回一个类型是```Prop```的实例。

最后我们来表达 $\forall n \in \mathbb{N} \sum_{i=1}^n i = \frac{n(n+1)}{2}$ 中命题部分 $\forall A, B$ 、 $C = D$，其中 $A$ 、$B$ 、 $C$ 、 $D$ 是之前表述过的部分，汇总如下。

|符号|含义|lean表示|
|-|-|-|
|$A$|$n\in \mathbb{N}$|```n : Nat```|
|$B$|$\sum_{i=1}^n i = \frac{n(n+1)}{2}$|```sum_zero_to_n_directly n = sum_zero_to_n_formula n```|
|$C$|$\sum_{i=1}^n i$|```sum_zero_to_n_directly n```|
|$D$|$\frac{n(n+1)}{2}$|```sum_zero_to_n_formula n```|

更进一步我们可以将完整表示列出

```{lean-playground} :height: 250px
def sum_zero_to_n_formula (n : Nat) : Nat := 
  n * (n + 1) / 2

def sum_zero_to_n_directly (n : Nat) : Nat := 
  if n = 0 then 0 else sum_zero_to_n_directly (n - 1) + n

def prop_eq_sum : Prop :=
  ∀ n : Nat, sum_zero_to_n_directly n = sum_zero_to_n_formula n

#check prop_eq_sum
```

与计算部分表达类似，将上面描述的所有部分按照顺序拼装在了一起，就可以定义函数出 ```prop_eq_sum``` 。它返回的正是命题：

$$
    \forall n\in \mathbb{N},\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

本身。

至此我们在lean中完成了命题的命题的表述。一般来说，在lean中，请将一个定义或实现拆解为尽可能小的组成部分。
确保一个定义或实现中只关心一件事情。
这样做是因为lean由于需要保证精确性，许多复杂的表述并不容易在证明处理过程中处理。
这有一点像在刚开始学习证明或计算时，老师可能会要求学生写出每一步所有使用的定理，并禁止省略任何步骤。
熟练之后则无需再注意这些规范，因为它已经成为了你的直觉的一部分。

至此终于使用lean完成了对命题的表述，下面终于可以开始问题的证明了。

### 证明命题

证明的书写总有固定的格式（无论会不会证，先写个证字把条件抄一遍再说）lean也不会例外。首先给出证明的基本样式：

```{lean-playground} :height: 250px
def sum_zero_to_n_formula (n : Nat) : Nat := sorry
def sum_zero_to_n_directly (n : Nat) : Nat := sorry

def prop_eq_sum : Prop :=
  ∀ n : Nat, sum_zero_to_n_directly n = sum_zero_to_n_formula n

theorem proof_eq_sum : prop_eq_sum := by
  sorry

```

这里出现了一些之前没有的关键字 ```sorry``` 、 ```by``` 和 ```theorem``` 。

#### ```theorem``` 关键字

表示这是一个定理，它后面会接着下面几个部分，

1. 定理名称：在这个例子中定理的名称叫 ```proof_eq_sum```
2. 定理需要的参数：它的位置在定理名称之后 ```:``` 之前，在这个例子中，定理不要额外的参数。
之后我们会看到需要参数的定理。
3. 定理结论：它是位于 ```:``` 于 ```:=``` 之间的部分。在这个定理就是我们希望证明的命题```prop_eq_sum```。在现阶段我们希望他是一个 ```Prop``` 的实例。
4. 证明过程：它位于 ```:=``` 之后。 在这个例子里是 ```by sorry``` ，
它由两个部分组成：一部分是 ```by``` 有点类似于证明开头的 ```证：``` 字。可以暂时理解成无论需要证明什么都需要遵循的固定格式。另一部分 ```sorry``` 表示我暂时不知道怎么证明它，写个东西占个位置。（有点类似于我觉得它是对的但地方太小了写不下）

#### ```by``` 关键字

可以简单理解成证明的一种固定格式。对于刚开始理解可以不去深究它。

>[策略模式（Tactic Mod）](https://lean-lang.org/theorem_proving_in_lean4/Tactics/#Theorem-Proving-in-Lean-4--Tactics)：```by``` 关键字实际上是策略模式的开关。
添加它表示我们使用策略模式进行证明。在策略模式中我们可以使用很多策略进行证明，如连续使用交换律、结合律的```ring```，对函数/定理按定义展开的```rewrite``` / ```rw```，等。
更多的策略可以查看[策略汇总](https://lean-lang.org/doc/reference/latest/Tactic-Proofs/Tactic-Reference/#tactic-ref)。这里不再详述。
>
>项式模式（Term Mode）:不添加则停留在项式模式（Term Mode）。它要求使用者手动的构造所有需要的东西。因此需要使用者熟悉所有可能的定理与其条件。这一般来说是一个非常高的要求。因此在实际使用之会使用 ```let``` 关键字手动在策略模式下使用它，以减少证明所需的复杂度。

#### ```sorry``` 关键字

它的用法就是在函数不知道怎么实现，或者证明不知道怎么证的时候用来占位置的符号。
在本例中函数里使用的 ```sorry``` 是为了节约篇章而省略重复的一种手段。```theorem``` 中出现的 ```sorry``` 则是暂时不知道如何证明的占位符。

#### 如何应用定理

在开始正式证明更复杂的东西之前，首先要学会如何应用已经存在的定理。
毕竟我们没有必要手搓所有的定理，而且这些定理很多并不好证明。稍有不慎就会遇上大量未曾理解或者不曾思考的问题。给我们的学习信心带来大量的打击。为了暂时回避这个事情。我们就要站在巨人肩膀上更进一步。为此首先来看一点简单的命题与证明。

```{lean-playground} :height: 250px
def prop_nat_comm_1 (n : Nat) (m : Nat) : Prop :=
  n + m = m + n

def prop_nat_comm_2 (n m : Nat) : Prop :=
  n + m = m + n

#check Nat.add_comm

theorem prof_not_finish (n : Nat) (m : Nat) : prop_nat_comm_1 n m := by
```

这里我们写了两个返回类型时命题的函数 ```prop_nat_comm_1``` 与 ```prop_nat_comm_1``` 。之后我们会给出两个定理，它们分别证明了这两个命题。在详细阅读证明之前，我们可以观察一下 ```prof_not_finish``` 这个证明```by``` 后面有红色波浪线。当鼠标放上去的时候就可以看到下面这一段输出：

```lean
unsolved goals
n m : Nat
⊢ prop_nat_comm_1 n m
```

这段输出表示lean在提示我们证明没有完成。
更详细的它列出了我们当前具有的条件（条件区域） ```n m : Nat``` ，它表明了我们有自然数 ```n``` 与 ```m``` 。
此时我们只有一个条件，因此只显示了一条。如果有很多条件也，它会一一列出。
后面 ```⊢ prop_nat_comm_1 n m``` 表示了定理希望证明的目标。
一般称为目标区域。此时我们也只有一个尚未证明的目标，因此只显示了一个。如果有更多证明目标它会在 ```Tactic state``` 中显示所有尚未完成的证明目标与目标名称。

更进一步 ```Tactic state``` 会动态展示证明过程进行到当前步所持有的状态——包括目标条件等。
可以在编辑器中移动光标去观察状态（如目标条件的变化），这有助于理解证明前后两步改变了什么。除了前面提到的的条件区域与目标区域等，它还包含了许许多多小功能，如绿色表示上一步证明将该式子改写为绿色区域的内容，红色表示当前步骤将要改写这段式子，等等。简而言之所有对证明所做的操作都会在这个区域中体现。

##### 定理 ```prof_1```

```{lean-playground} :height: 250px
def prop_nat_comm_1 (n : Nat) (m : Nat) : Prop :=
  n + m = m + n

theorem prof_1 (n : Nat) (m : Nat) : prop_nat_comm_1 n m := by
  -- goal prop_nat_comm_1 n m
  rw [prop_nat_comm_1]
  -- goal n + m = m + n
  have eq1 : n + m = m + n := Nat.add_comm n m
  exact eq1
```

它的声明部分是

```lean
theorem prof_1 (n : Nat) (m : Nat) : prop_nat_comm_1 n m
```

表示了有一个定理名叫 ```prof_1``` ，它对于给定的 ```Nat``` 实例 ```n``` 、 ```m``` ， 有调用 ```prop_nat_comm_1 n m``` 后返回的命题成立。
它与我们之前写出的命题 [```proof_eq_sum```](#证明命题) 不同。

> ```lean
> theorem proof_eq_sum : prop_eq_sum := by sorry
> ```

这个定理接受两个 ```Nat``` 类型的参数 ```n``` 、 ```m```， 而 ```proof_eq_sum``` 并不接受参数。
这是一个非常关键的区别，这个参数为我们如何引用他人的定理奠定了基础。
在随后的证明过程中我们可以更进一步的看到如何引用他人定理。

使用自然语言表达这个定理有

> 定理 ```prof_1```：对于给定的自然数 $n$ 、 $m$ 。 有 ```prop_nat_comm_1 n m``` 成立。
> 其中 ```prop_nat_comm_1 n m``` 表示命题：对于给定的自然数 $n$ 、 $m$ 有 $n+m=m+n$ 。

这个表述略微有些不太符合阅读习惯，因为人们会很自然的把 ```prop_nat_comm_1 n m``` 展开为对应的命题。从而表述为

> 定理 ```prof_1```：对于给定的自然数 $n$ 、 $m$ 。 有 $n+m=m+n$ 成立。

在表述定理时并不会保留符号 ```prop_nat_comm_1```。
在lean语言中，这样的展开过程叫做 ```rw``` 或 [```rewrite```](https://lean-lang.org/doc/reference/latest/The-Simplifier/Rewrite-Rules/)。```rw``` 这个策略的含义就是在目标中将某个符号按照定义进行改写。在 ```prof_1``` 的证明中我们可以看到 ```rw [prop_nat_comm_1]``` 将目标 ```prop_nat_comm_1 n m``` 改写为 ```n + m = m + n``` , 它就是 ```prop_nat_comm_1``` 的应定义部分。

完成这个改写之后，发现目标已经变为了 ```n + m = m + n``` 。 这时你一定会有一个想法“这也要证？这不就是加法交换律么？”。
是的没错，这确实需要证明，我们觉得它成立是因为我们已经学习过交换律，并且知道对于加法它成立。
但是对于计算机来说，它很笨，它什么都不知道。因此我们需要去从标准库中寻找加法交换律并应用它。
大部分从直觉上觉得十分显然的东西，标准库里都会有（如果没有则可能时这东西不成立，或者说成立的条件与我们预想的不一致。这往往是十分复杂的。）。从而写出

```lean
have eq1 : n + m = m + n := Nat.add_comm n m
```

> ```have``` 关键字： 它表示应用某个定理的格式。后面一般接三个部分
>
> 1. ```:=``` 之后的表示应用的定理。 这里我们需要对 ```n``` 、```m``` 应用交换律 ```Nat.add_comm``` ，即 ```Nat.add_comm n m``` 。
> 2. ```:``` 之后， ```:=``` 之前的部分是定理应用后的结论。
> 在这里我们希望得到的结论是：交换律 ```n + m = m + n``` 成立。这一部分可以省略不写。但建议写出来，这可以明确你的思路，以防止自动推断失败，与推断出的并不是你想要的。
> 3. ```have``` 之后， ```:``` 之前的部分是名称。它代指了对这个定理应用的结果。起一个名称方便后续使用。在这个例子中我们起名叫 ```eq1``` 代指对加法交换律的应用结果

其中标准库中 [```Nat.add_comm```](https://leanprover-community.github.io/mathlib4_docs/Init/Data/Nat/Basic.html#Nat.add_comm) 的声明如下

```lean
theorem Nat.add_comm (n m : Nat) : n + m = m + n
```

可以看出它返回的命题就是前面我们希望的得到的结论 ```n + m = m + n```。 
在应用交换律之后我们发现此时的目标并没有改变。而条件区域多处了一个 ```eq1 : n + m = m + n``` 。

```lean
1 goal
n m : ℕ
eq1 : n + m = m + n
⊢ n + m = m + n
```

也就是说这时我们已经很接近结论了， 只要说明 ```eq1``` 就是要证的目标就可以了。这一句在 lean 中表达为 

```lean
exact eq1
```

> ```exact``` 策略：它通常用于处理存在性命题。表明已经有一个东西就是我们想要的。它只有一个部分
>
> - ```exact``` 后边的部分，强调表示一些东西存在。
>
> 在这里我们的目标是 ```⊢ n + m = m + n``` ，而恰好 ```eq1``` 就是它。

小结一下，对于 ```proof_1``` 我们使用了策略 ```rewrite``` 来展开定义。
使用 ```have``` 关键字来应用标准库中的定理。
使用 策略 ```exact``` 来说明存在交换律。
可以看出 lean 语言的证明过程就是逻辑证明过程的延续。在后面的例子中我们可以更加明确观察到这一点。

##### 定理 ```prof_2```

```{lean-playground} :height: 250px
def prop_nat_comm_2 (n m : Nat) : Prop :=
  n + m = m + n

theorem prof_2 (p : Nat) (q : Nat) : prop_nat_comm_2 p q := by
  -- goal prop_nat_comm_2 p q
  rw [prop_nat_comm_2]
  -- goal p + q = q + p
  rw [Nat.add_comm]
```

观察 ```prof_2``` 的声明，

```lean
theorem prof_2 (p : Nat) (q : Nat) : prop_nat_comm_2 p q 
```

容易发现它与 ```prof_1``` 并没有非常大的区别，只是将参数 ```m``` 、 ```n``` 换为了 ```p``` 、 ```q``` 。
因此它的证明也是类似的

```lean
  -- goal prop_nat_comm_2 p q
  rw [prop_nat_comm_2]
  -- goal p + q = q + p
  rw [Nat.add_comm]
```

第一步依然是使用 ```rewrite``` 展开定义将目标 从 ```prop_nat_comm_2 p q``` 转化为 ```p + q = q + p``` 。而第二步有所不同，直接使用已经证明过的 ```Nat.add_comm``` 改写目标。证明就完毕了。实际上这里发生了相当多的过程。

1. ```rewrite``` 会根据你的提示对 ```p``` 、```q``` 使用 ```Nat.add_comm``` ，得到 ```p + q = q + p``` 的实例，这里这个实例并没有像 ```proof_1``` 中被赋予了一个名字 ```eq1``` 。而是一个匿名的存在。
2. 之后  ```rewrite``` 发现 ```p + q = q + p``` 的实例就是目标，从而完成了证明。相当于自动地完成了 ```exact eq1``` 。

从这个例子里可以看出 ```rewrite``` 策略是一个语法糖，它是由更基础的 ```have``` 、 ```exact``` 等，构成的。这也是策略（tactic）称呼的由来，即包装一些高频操作。在 lean 中只要跟随常规的证明思路将其翻译成 lean 语言的语法形式就好。不必可以追求使用策略或回避使用各种策略。

此外需要注意的是 lean 语言的交互过程是不断地转换目标，直到变成某个条件。证明就结束了。类似于

> 1. 为了证明目标1，只要证目标2。 
> 2. 为了证名目标2，只要证墓表3.
> 3. ......
> 4. 这个目标就是条件，证毕。

这更符合人们思考与分析证明的习惯。而常规的证明过程是将思考倒置，从条件出发逐渐构造结论的过程。因此经常出现“注意到”这样完全注意不到的过程。这就是分析证明与书写证明的区别。因此 lean 证明的表达方式更容易理解，也有助于理解为什么这里要做这个操作。

\begin{aligned}
&\begin{aligned}
 &0&+&1    &+&2    &+&...&+&(n-2)&+&(n-1)&+&n\\
+&n&+&(n-1)&+&(n-2)&+&...&+&2    &+&1    &+&0\\

## 类型<->命题、实例<->证明

## abc

theorem eq_symm (A : Type) (n m : A) (h : n = m) : m = n := by
    let h0 := @Eq.rec
    let h1 := @h0 A
    let h2 := @h1 n
    let h3 := @h2 (fun (y' : A)=> fun (h : n = y' ) => y' = n )
    let h4 := @h3 (Eq.refl n)
    let h5 := @h4 m
    let h6 := @h5 h
    exact h6

```

机器定理证明的价值主要体现在三方面：一是**绝对的可靠性**——一旦证明通过内核检查，正确性几乎等同于逻辑公理本身的可信度；二是**大规模数学的可维护性**——像 `mathlib` 这样的形式化库已经积累了数十万条定理，可被组合复用；三是**软硬件验证**——CompCert 编译器、seL4 微内核、Intel 处理器算法等关键系统都依赖此类工具给出形式化正确性保证。近年来 Kevin Buzzard 等人发起的「把现代数学搬进 Lean」运动，以及 Terence Tao 用 Lean 形式化自己最新论文的实践，让机器定理证明从计算机科学的小众工具，逐渐成为数学研究本身的一部分。

本指北将以 **Lean 4** 为载体，从「类型即命题、程序即证明」这一 Curry–Howard 视角出发，带你逐步上手机器定理证明。

对于人来说，证明写成这个样子基本就够用了。



换言之，在大部分场合下人们都认为这个证明过程证明了上述命题。
但对于机器来说这里有好多模棱两可的地方，比如

1. $n=1.5$行不行，$n=-1$行么？
2. $0+1+2+...+n = n+(n-1)+(n-2)+...+0$ 怎么就成立了？(小朋友经常问这个问题)

面对这些问题，肯定有人要问这不是抬杠么。$n$当然是个自然数，加法交换律成立，$0+1+2+...+n$当然等于$n+(n-1)+(n-2)+...+0$。 但实际上这些被我们认知为理所当然的东西，是我们经年累月的训练的得到的直觉（毕竟小朋友对交换律的掌握并不熟练，就会觉得并不显然）。
但很不幸的是计算机是一个非常傻的东西。不告诉它$n$是自然数，它会不知道$n$是什么。
不说明 (2) 成立它就是不知道这是连续应用加法交换律的结果。
换言之好多人觉得十分显然的东西，计算机它觉得完全不显然。因此将条件补充完整后得到命题
