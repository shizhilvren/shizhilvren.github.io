# 机器定理证明

**机器定理证明（Automated / Interactive Theorem Proving）** 或者说“使用计算机来辅助证明定理”，是指借助计算机来书写、检验乃至自动发现（并证明）数学证明的一类技术。
它的核心思想与“使用计算机来辅助计算”（如各种编程语言C/C++、Rust）相似，都是将自然语言描述的计算/证明过程转化另一种编程语言的技术。
使得我们可以借助计算机的运算能力与精确性来高效、准确的完成计算和证明。

按照人与机器的分工，大致分为两类：

- **自动定理证明（ATP, Automated Theorem Proving）**：机器自主搜索证明，典型代表是 SAT/SMT 求解器（如 Z3、CVC5）和一阶逻辑证明器（如 E、Vampire）。它们擅长有限/可判定片段，但难以处理深奥的数学。这不是本文的重点。
- **交互式定理证明（ITP, Interactive Theorem Proving）**：由人给出证明大纲、计算机负责检查每一步并补全细节，代表系统包括 **Lean**、Coq/Rocq、Isabelle/HOL、Agda 等。它能够覆盖任意复杂的现代数学。

本文主要以 [lean](https://leanprover-community.github.io/learn.html) 语言为引子来介绍交互式定理证明。
并带领读者实际编写一些证明，体验编写证明的全部流程，其目标是让读者了解 lean 语言下的基本范式与概念。

## 使用计算机来辅助计算

在开始使用lean证明各种东西之前，我们先看一个“使用计算机来辅助计算”的例子。 这有助于理解“使用计算机来辅助证明定理”。

### 计算$1+2+3+...+100=?$

如果我们想计算

$$
0+1+2+...+100=?
$$

我们可以在不借助任何计算工具的条件下写出的计算过程

$$
\begin{aligned}
&0+1+2+3+...+100 \\
=&(0+1)+2+3+...+100 \\
=&(1+2)+3+...+100 \\
=&(3+3)+...+100 \\
=&... \\
=&5050
\end{aligned}
$$

这个计算过程非常的直观，就是从左到右一个个数字加过去。朴实而原始。那么如果考虑“使用计算机来辅助计算”，可以得到如下程序

```c++
unsigned int sum_zero_to_one_hundred() {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= 100; i++) {
        sum = sum + i;
    }
    return sum;
}
```

这个函数编写的非常的直观，直接将“人肉计算”过程翻译成了计算机程序。
但是这实际上是一个非常了不起的过程，它完成了从“人肉计算”到“使用计算机来辅助计算”的跨越。
计算机代替我们完成了繁琐的计算过程。

### 计算$0+1+2+...+n=?$

更进一步如果将问题改为

$$0+1+2+...+n=?$$

这时如果继续尝试进行人肉计算，则会得到下面的计算过程：

$$
\begin{aligned}
&0+1+2+...+n \\
=&1+2+...+n \\
=&3+...+n \\
=&... \\
\end{aligned}
$$

但却遇到了一个问题，并不知道该加到多少停止。这是因为这个问题中并没有给出 $n$ 的具体数值。因此只能说

$$请依照加法的计算方法一个个数字加过去，直到加到了 n 时停止$$

当然如果继续“使用计算机来辅助计算”，依然可以写出如下函数

```c++
unsigned int sum_zero_to_n_directly(unsigned int n) {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= n; i++) {
        sum = sum + i;
    }
    return sum;
}
```

“使用计算机来辅助计算”并不在乎是否给出了一个包含具体数值的 $n$ 。因为它只是上面“人肉计算”的一种翻译。这个程序只是将“请依照加法的计算方法一个个数字加过去，直到加到了 $n$ 时停止”这句话翻译成了计算机理解的形式。
当然经过一些思考还可以写出

```c++
unsigned int sum_zero_to_n_formula(unsigned int n) {
    unsigned int sum = n * (n + 1) / 2;
    return sum;
}
```

这个计算函数就有些不同了。虽然计算结果依然相同，但计算过程则完全不一致。
因为它不再是“人肉计算"的简单翻译结果。
而是使用了更高级的结论简化了计算过程。具体来说是应用了命题：

$$
    0+1+2+...+n = \frac{n(n+1)}{2}
$$

这个结论非常的基本，以至于我们我们不会怀疑它的正确性。毕竟这已经是书本上学过并验证过的的东西了。

## 使用计算机来辅助证明

接下来就以求和公式为例演示如何“使用计算机来辅助证明”。
首先回顾一下在自然语言中我们是如何证明这一命题的呢？
其中一种方式是直接使用数学归法。

**命题**： 对任意的自然数 $n$ ，有 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立

**证**：对的自然数 $n$ 使用数学归纳法

1. 当 $n=0$ 时，有 $0=\frac{0\times (0+1)}{2}$ ，成立。
2. 假设 $n=k$ 时，结论成立，即 $0+1+2+...+k = \frac{k(k+1)}{2}$ 成立。下证明 $n=k+1$ 时有 $0+1+2+...+k+(k+1) = \frac{(k+1)(k+2)}{2}$ 成立。直接计算有

    $$
    \begin{aligned}
          &0+1+2+...+k+(k+1) \\
         =&(0+1+2+...+k)+(k+1)&加法结合律\\
         =&\frac{k(k+1)}{2}+(k+1)&使用归纳假设\\
         =&\frac{k(k+1)+2(k+1)}{2}&通分\\
         =&\frac{(k+2)(k+1)}{2}&提取公因式\\
         =&\frac{(k+1)(k+2)}{2}&乘法交换律\\
    \end{aligned}
    $$

    因此 $n=k+1$ 时结论成立。

    由(1)(2)与归纳法知 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立。$\square$

下面将这个由自然语言描述的证明过程翻译为计算机理解的方式。

### 描述问题

一般来说，使用自然语言描述问题有太多的模糊性。因此会使用数学语言来描述问题，以减少自然语言带来的模糊性。进而将命题改写为如下形式。

**命题**：

$$
    \forall n\in \mathbb{N},\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

其中 $\mathbb{N}$ 表示自然数的集合。 
> 注：将自然语言描述的命题翻译为数学语言描述的命题这一过程称为问题的形式化，其目的是在于消除表达的歧义。
当然它也会带来副作用，在训练不足时会觉得这种表述非常别扭且难以理解。
这是正常的，因为形式化表述本身对于熟悉自然语言表述的人来说并不是自然而然的。
不如说任何表述方式或者规范本身都有其门槛和表达范式。

观察这个命题可以发现它是由命题部分和计算部分组成。
首先拆解第一个命题部分

$$
\forall A, B
$$

其中 $A$ 是 $n \in \mathbb{N} $， $B$ 是 $\sum_{i=1}^n i = \frac{n(n+1)}{2}$ 。
它描述的是对于任意的 $A$ 条件成立则 $B$ 结论成立。一般我们将其读作对任意的 $A$ 则 $B$ 。

再拆解 $B$ 这个部分

$$
\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

它由下面三个部分组成，

$$
\begin{aligned}
\sum_{i=1}^n i &&  \frac{n(n+1)}{2} && C = D
\end{aligned}
$$

其中 $\sum_{i=1}^n i$ 、 $\frac{n(n+1)}{2}$ 都是一个计算过程， 而 $C=D$ 是一个命题。

综上，我们得到了命题部分：

$$
\begin{aligned}
 \forall A, B && C = D
\end{aligned}
$$

和计算部分

$$
\begin{aligned}
\sum_{i=1}^n i &&  \frac{n(n+1)}{2}
\end{aligned}
$$

#### 表述计算部分

下面首先使用 lean 表述计算部分 $\frac{n (n + 1) }{ 2}$

```{lean-playground} :height: 200px
def sum_zero_to_n_formula (n : Nat) : Nat :=
    n * (n + 1) / 2

#eval sum_zero_to_n_formula 100
```

其中 ```sum_zero_to_n_formula``` 表示了计算部分 $\frac{n(n+1)}{2}$ 。
它被定义成一个函数。在 lean 中的一个函数由下面几个部分组成：

1. 关键字```def```表示声明一个函数（计算部分）。这是lean语法的一部分。
2. 紧跟在 `def` 后面的是函数的名称，在本例中是 `sum_zero_to_n_formula` 。
3. 名称之后 `:` 之前的是参数部分。在本例中是 ```(n : Nat)``` 。
它表示 ```sum_zero_to_n_formula``` 接受一个类型为 ```Nat``` 名称叫做 ```n``` 的参数。```Nat``` 类型在这里表示自然数。
4. ```:```与```:=```中间的```Nat```表示函数```sum_zero_to_n_formula```的返回类型为一个自然数类型。
5. ```:=```后面表示的是函数的实现。返回 ```n * (n + 1) / 2``` 的计算结果。这里返回实例并不需要写 ```return``` 关键字，这是与 C/C++ 不同的地方。

我们使用```#eval```这个方法来调用这个函数。在代码右侧的消息（Message）中可以看到它的输出 ```5050``` 代表着从$0$加到$100$的结果是$5050$。

接下来来表述另一个计算部分 $\sum_{i=1}^n i$

```{lean-playground} :height: 200px
def sum_zero_to_n_directly (n : Nat) : Nat :=
  if n = 0 then 
    0 
  else 
    sum_zero_to_n_directly (n - 1) + n

#eval sum_zero_to_n_directly 100
```

可以看到函数 `sum_zero_to_n_directly` 使用的是递归定义：

$$
f(n) =
\begin{cases}
0, & n = 0 \\
f(n-1) + n, & n = 1,2,3,...
\end{cases}
$$

其中 $f$ 表示了函数 $sum\_zero\_to\_n\_directly$ 。
这是因为在lean中， $\sum_{i=1}^n i$ 如果表述成像C函数一样包含循的直接表达形式会给证明带来麻烦，因此这里选用了证明更加容易的递归表达式。
换言之，lean 的表达范式是推荐优先写成递归形式的。
这一点与其它编程语言刚好相反，在其它语言中通常会使用专门的技巧尽可能的消除递归表达。而在lean中请尽量保留递归表达。

#### 表述命题部分

在表述整个复杂命题之前，首先来看一些基础的命题与其对应在 lean 中表述。

##### 不含参数的命题表述

下面给出了4个命题

**命题**:

$$
\begin{aligned}
    1+1&=2\\
    1+1&=3\\
    \forall n\in \mathbb{N}, n &+ 1 = 4\\
    \exists n\in \mathbb{N}, n &+ 1 = 4\\
\end{aligned}
$$

```{lean-playground} :height: 300px
def prop_one_add_one_eq_two : Prop :=
  1 + 1 = 2

def prop_one_add_one_eq_three : Prop :=
  1 + 1 = 3

def prop_n_add_one_eq_four : Prop :=
  ∀ n : Nat, n + 1 = 4

def prop_exists_n_add_one_eq_four : Prop :=
  ∃ n : Nat , n + 1 = 4

```

可以看到命题的表述与之前的计算函数类似。更准确的说是有一个函数，它的返回类型为命题。
它与之前的函数最大的不同它的返回类型不再是```Nat```（自然数），而是一个新类型```Prop```(propositions，命题)。
而实例 ```1 + 1 = 2``` 、```1 + 1 = 3``` 、```∀ n : Nat, n + 1 = 4``` 和 ```∃ n : Nat , n + 1 = 4``` 都是类型```Prop``` 的实例。就如同实例 ```1``` 、```2``` 和 ```3``` 都是类型 ```Nat``` 的实例一样。

> **实例与类型**：
> 为了表述方便，一般使用下面的记法表达实例与类型的关系。
>
> $$\begin{aligned} 1 :& Nat\\ 1 + 1 = 3 : &Prop\\∃ n : Nat , n + 1 = 4 : &Prop \end{aligned}$$
>
> 其中 ```:``` 左边表示实例，右边表示类型。
> 分别读作
>
> $$\begin{aligned} 1 &是Nat的实例\\1+1=3 &是Prop的实例\\存在n是Nat的实例，使得n+1=4 &是Prop 的实例 \end{aligned}$$
>
> 其中前两个比较简单，第三个则相对复杂，因为它涉及了嵌套关系。它嵌套了下面两个实例于类型的关系
>
> $$\begin{aligned} n : &Nat\\ ∃  \_ , n + 1 = 4 : &Prop \end{aligned}$$
>
>将 ```∃ n : Nat , n + 1 = 4 ```这个东西视为一个整体才是```Prop```的实例。
>也就是说```存在n是Nat的实例，使得n+1=4```作为一个整体才是```Prop```的实例。
>单独拿出任何一部分都不是这个式子中期望表达的Prop的实例。
>
>>注：实际上 ```Prop``` 这个类型并没有什么特殊的意义，它和众多类型如 ```Nat``` 没有区别。只是使用lean语言的人觉得这个东西可以被证明，因此单独拿出来讨论。

对于前两个命题，命题：$1+1=2$ 和命题 $1+1=3$。显然我们知道命题： $1+1=2$ 是真命题。命题： $1+1=3$ 是假命题。
在lean中```Prop```不会限制一个假命题的表达。即命题无论真假都可以被表述。当然假命题的证明在 lean 中是表述不出来的。
lean 语言会帮助我们完成证明正确性的检验，而使用自然语言表述的证明需要人进行检查。
因此如果有个人一本正经的胡说八道，去分析他所写的证明是对是错也是非常费时费力的。
在lean 中就变得非常简单了，只需要直接检查一下编译器是否提示了错误即可。

对于后两个命题，命题： $\forall n\in \mathbb{N}, n + 1 = 4$ 和命题： $\exists n\in \mathbb{N}, n + 1 = 4$ ，它们分别表述了任意与存在。lean是数学表述的直接翻译。只是将 $\in$ 替换为了 $:$ ，其它部分并没有变化。

|lean描述|数学描述|
|-|-|
|```∀ n : Nat```|$\forall n\in \mathbb{N}$|
|```∃ n : Nat```|$\exists n\in \mathbb{N}$|

> 注：严格来说这个表述不够严谨，因为```Nat```类型并非一个集合，```n : Nat``` 表述的是 $n$ 是 ```Nat``` 的一个实例$n$。
```n ∈ (setOf fun _ : Nat => true)``` 表达的才是 $n$ 属于的由```Nat```所有元素构成的集合。
这两者是完全等价的，因此本文选用更易于理解的第一种表述。

##### 含有参数的命题表述

接下来再来看另一组返回命题的函数

```{lean-playground} :height: 350px
def prop_nat_symm_1 (n : Nat) (m : Nat) : Prop :=
  n + m = m + n

def prop_nat_comm_2 (n m : Nat) : Prop :=
  n + m = m + n

def prop_nat_symm_3 : Prop :=
  ∀ n : Nat, ∀ m : Nat, n + m = m + n

def prop_nat_symm_4 : Prop :=
  ∀ n m : Nat, n + m = m + n

#check prop_nat_symm_1 
#check prop_nat_symm_1 1 2 
#check prop_nat_symm_4
```

其中(1)(2)是一组，命题(3)(4)是另一组。
其中(1)(2)这一组，描述的是有一个函数接受两个类型为 ```Nat``` 的参数，名叫 $n$ 与 $m$ ，返回命题 $n + m = m + n$ 。如果翻译为自然语言则是

命题：

$$
    对于给定的自然数n 、m, 有 n+m=m+n
$$

强调的是如果从外部传递给函数内部 $n$ 和 $m$ 则还给函数返回命题 $n+m=m+n$ ，当然如果从外部传递的是 $p$ 和 $q$ 则函数返回命题 $p+q=q+p$。 
至于参数 $n$ ，$m$ 从哪里来这个函数并不关心，这一部分由构造命题的人从外部传入。

而命题(3)(4)描述的就是命题：

$$
    \forall n,m \in \mathbb{N},n+m=m+n
$$

它不依赖任何外部传入的参数，直接了当的说明想表达的命题。

在这里我们使用了 ```#check``` 来检查某个实例的类型，在lean中它与 ```#eval``` 一样会显示一些消息在message中。其中

1. ```#check prop_nat_symm_1``` 显示了 ```prop_nat_comm_1 (n m : ℕ) : Prop``` ，其含义是 ```prop_nat_comm_1``` 接受了两个参数 ```n m : ℕ``` 后返回一个的类型是```Prop```的实例。
2. ```#check prop_nat_symm_1 1 2``` 显示了 ```prop_nat_comm_1 1 2 : Prop``` ，其含义是 ```prop_nat_comm_1``` 接受了两个参数 ```1``` 与 ```2``` 后返回一个类型是```Prop```的实例。
换言之 `#check prop_nat_symm_1 1 2` 整体是 `Prop` 的实例。
实际上我们还知道该实例就是 ```1+2=2+1``` ，当然```#check``` 并不显示这些。
3. ```#check prop_nat_symm_4``` 显示了 ```prop_nat_comm_4 : Prop``` ，其含义是 ```prop_nat_comm_4``` 不需要任何输入就返回一个类型是```Prop```的实例。

最后来表达 $\forall n \in \mathbb{N} ,\sum_{i=1}^n i = \frac{n(n+1)}{2}$ 中命题部分 $\forall A, B$ 与 $C = D$，其中 $A$ 、$B$ 、 $C$ 、 $D$ 是之前表述过的部分，汇总如下。

|符号|含义|lean表示|
|-|-|-|
|$A$|$n\in \mathbb{N}$|```n : Nat```|
|$C$|$\sum_{i=1}^n i$|```sum_zero_to_n_directly n```|
|$D$|$\frac{n(n+1)}{2}$|```sum_zero_to_n_formula n```|
|$B$|$\sum_{i=1}^n i = \frac{n(n+1)}{2}$|```sum_zero_to_n_directly n = sum_zero_to_n_formula n```|

更进一步的，可以将整个命题完整表述出来

```{lean-playground} :height: 250px
def sum_zero_to_n_formula (n : Nat) : Nat := 
  n * (n + 1) / 2

def sum_zero_to_n_directly (n : Nat) : Nat := 
  if n = 0 then 0 else sum_zero_to_n_directly (n - 1) + n

def prop_eq_sum : Prop :=
  ∀ n : Nat, sum_zero_to_n_directly n = sum_zero_to_n_formula n

#check prop_eq_sum
```

与计算部分表达类似，将上面描述的所有部分按照顺序拼装在了一起，就可以定义函数出 ```prop_eq_sum``` 。它返回的正是命题：

$$
    \forall n\in \mathbb{N},\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

本身。

至此将要证明的命题在 lean 中得到了完整的表述。一般来说，在 lean 中，请将一个定义或实现拆解为尽可能小的组成部分。
以确保一个定义或实现里只关心一件事情。
这样做是因为 lean 由于需要保证精确性，许多复杂表述在证明过程中的处理会变得十分困难。
这有一点像在刚开始学习证明或计算时，老师可能会要求学生写出每一步所使用的定理名称，并禁止省略任何步骤。
熟练之后则无需再注意这些规范，因为它已经成为了你的直觉的一部分。

至此终于使用lean完成了对命题的表述，下面终于可以开始问题的证明了。

### 证明命题

证明的书写总有固定的格式（无论会不会证，先写个证字把条件抄一遍再说）lean也不会例外。

#### 证明命题的基本格式

下面展示了一段证明的基本样式：

```{lean-playground} :height: 250px
def sum_zero_to_n_formula (n : Nat) : Nat := sorry
def sum_zero_to_n_directly (n : Nat) : Nat := sorry

def prop_eq_sum : Prop :=
  ∀ n : Nat, sum_zero_to_n_directly n = sum_zero_to_n_formula n

theorem proof_eq_sum : prop_eq_sum := by
  sorry

```

这里出现了一些之前没有的关键字 ```sorry``` 、 ```by``` 和 ```theorem``` 。

#### ```theorem``` 关键字

表示这是一个定理，它后面会紧接着如下几个部分，

1. 定理名称：在这个例子中定理的名称叫做 ```proof_eq_sum```
2. 定理需要的参数：它的位置在定理名称之后 ```:``` 之前，在这个例子中，定理不要额外的参数。
之后我们会看到需要参数的定理。
3. 定理结论：它是位于 ```:``` 于 ```:=``` 之间的部分。在这个定理就是我们希望证明的命题```prop_eq_sum```。在现阶段我们希望他是一个 ```Prop``` 的实例。
4. 证明过程：它位于 ```:=``` 之后。 在这个例子里是 ```by sorry``` ，
它由两个部分组成：一部分是 ```by``` 有点类似于证明开头的 ```证：``` 字。可以暂时理解成无论需要证明什么都需要遵循的固定格式。另一部分 ```sorry``` 表示我暂时不知道怎么证明它，写个东西占个位置。（有点类似于我觉得它是对的但地方太小了写不下）

#### ```by``` 关键字

可以简单理解成证明的一种固定格式。对于刚开始理解可以不去深究它。

>[**策略模式（Tactic Mod）**](https://lean-lang.org/theorem_proving_in_lean4/Tactics/#Theorem-Proving-in-Lean-4--Tactics)：```by``` 关键字实际上是策略模式的开关。
添加它表示后续证明使用策略模式进行。在策略模式中证明可以使用很多策略进行，如连续使用交换律、结合律的```ring```，对函数/定理按定义展开的```rewrite``` / ```rw```，等。
更多的策略可以查看[策略汇总](https://lean-lang.org/doc/reference/latest/Tactic-Proofs/Tactic-Reference/#tactic-ref)。这里不再详述。
>
>**项式模式（Term Mode）**：不添 `by` 关键字额加则停留在项式模式（Term Mode）。
> 它要求使用者手动的构造所有需要的实例。因此需要使用者熟悉所有可能的定理与其条件。
> 这一般来说是一个非常高的要求。因此在实际情况中会使用 ```let``` 或 `have` 关键字手动在策略模式下使用它，以减少证明所需的复杂度。

#### ```sorry``` 关键字

它的用法就是在函数不知道怎么实现，或者证明不知道怎么证的时候用来占位置的符号。
在本例中函数里使用的 ```sorry``` 是为了节约篇章而省略重复的一种手段。```theorem``` 中出现的 ```sorry``` 则是暂时不知道如何证明的占位符。

#### 如何应用定理

在开始正式证明更复杂的东西之前，首先要学会如何应用已经存在的定理。
毕竟完全没有必要手搓所有的定理，而且这些定理很多并不好证明。稍有不慎就会遇上大量未曾理解或者不曾思考的问题。给我们的学习信心带来大量的打击。为了暂时回避这个事情，就要站在巨人肩膀上更进一步。为此首先来看一点简单的命题与证明。

```{lean-playground} :height: 250px
def prop_nat_comm_1 (n : Nat) (m : Nat) : Prop :=
  n + m = m + n

def prop_nat_comm_2 (n m : Nat) : Prop :=
  n + m = m + n

#check Nat.add_comm

theorem proof_not_finish (n : Nat) (m : Nat) : prop_nat_comm_1 n m := by
```

这里我们写了两个返回类型为命题的函数 ```prop_nat_comm_1``` 与 ```prop_nat_comm_1``` （它们就是前文分析过命题）。之后会给出两个定理，它们分别证明了这两个命题。在详细阅读证明之前，我们可以观察一下 ```proof_not_finish``` 这个证明，它在 ```by``` 后面有红色波浪线。当鼠标放上去的时候就可以看到下面这一段输出：

```lean
unsolved goals
n m : Nat
⊢ prop_nat_comm_1 n m
```

这段输出表示lean在提示当前证明没有完成。
更详细的它列出了证明进行到此处具有的条件（条件区域） ```n m : Nat``` ，它表示了证明进行到此处拥有有自然数 ```n``` 与 ```m``` 。
此时它只有一个条件，因此只显示了一条。如果有很多条件也，它会一一列出。
后面 ```⊢ prop_nat_comm_1 n m``` 表示了定理希望证明的目标。
一般称为目标区域。此时证明进行到此处也只有一个尚未证明的目标，因此只显示了一个。
如果有更多证明目标它会在 ```Tactic state``` 中显示所有尚未完成的证明目标与目标名称。
后文会看到这样的例子，并完成每个目标的证明。

更进一步 ```Tactic state``` 会动态展示证明过程进行到当前步所持有的状态——包括目标、条件等。
可以在编辑器中移动光标去观察状态（如目标或条件的变化），这有助于理解证明前后两步改变了什么。
除了前面提到的的条件区域与目标区域之外，它还包含了许许多多小功能，如绿色表示上一步证明将该式子改写为绿色区域的内容，红色表示当前步骤将要改写这段式子，等等。简而言之对证明所做的所有操作都会在这个区域中体现。

##### 定理 ```prof_1```：交换律的一种证明

首先给出命题 `prop_nat_comm_1 n m` 的完整证明，随后逐行分析

```{lean-playground} :height: 250px
def prop_nat_comm_1 (n : Nat) (m : Nat) : Prop :=
  n + m = m + n

theorem prof_1 (n : Nat) (m : Nat) : prop_nat_comm_1 n m := by
  -- goal prop_nat_comm_1 n m
  rw [prop_nat_comm_1]
  -- goal n + m = m + n
  have eq1 : n + m = m + n := Nat.add_comm n m
  exact eq1
```

它的声明部分是

```lean
theorem prof_1 (n : Nat) (m : Nat) : prop_nat_comm_1 n m
```

表示了有一个定理名叫 ```prof_1``` ，它对于给定的 ```Nat``` 实例 ```n``` 、 ```m``` ， 有调用 ```prop_nat_comm_1 n m``` 后返回的命题成立。
它与之前写出的命题 [```proof_eq_sum```](#证明命题) 不同。

> ```lean
> theorem proof_eq_sum : prop_eq_sum := by sorry
> ```

这个定理接受两个 ```Nat``` 类型的参数 ```n``` 、 ```m```， 而 ```proof_eq_sum``` 并不接受参数。
这是一个非常关键的区别，这个参数为如何引用他人已经证明过的定理奠定了基础。
在随后的证明过程中可以更进一步的看到如何引用他人的定理。

使用自然语言表达这个定理有

> 定理 ```prof_1```：对于给定的自然数 $n$ 、 $m$ 。 有 ```prop_nat_comm_1 n m``` 成立。
> 其中 ```prop_nat_comm_1 n m``` 表示命题：对于给定的自然数 $n$ 、 $m$ 有 $n+m=m+n$ 。

这个表述略微有些不太符合阅读习惯，因为人们会很自然的把 ```prop_nat_comm_1 n m``` 展开为对应的命题。从而表述为

> 定理 ```prof_1```：对于给定的自然数 $n$ 、 $m$ 。 有 $n+m=m+n$ 成立。

在表述定理时并不会保留符号 ```prop_nat_comm_1```。
在lean语言中，这样的展开过程叫做 ```rw``` 或 [```rewrite```](https://lean-lang.org/doc/reference/latest/The-Simplifier/Rewrite-Rules/)。```rw``` 这个策略的含义就是在目标中将某个符号按照定义进行改写。
在 ```prof_1``` 的证明中可以看到第一步 ```rw [prop_nat_comm_1]``` 将目标 ```prop_nat_comm_1 n m``` 改写为 ```n + m = m + n``` , 而它就是 ```prop_nat_comm_1``` 的定义部分。

完成这个改写之后，可以发现目标已经变为了 ```n + m = m + n``` 。 这时你一定会有一个想法“这也要证？这不就是加法交换律么？”。
是的没错，这确实需要证明，人们觉得它成立是因为我们已经学习过交换律，并且知道对于加法它成立。
但是对于计算机来说，它很笨，它什么都不知道。因此我们需要去从标准库中寻找加法交换律并应用它。
大部分从直觉上觉得十分显然的东西，标准库里都会有（如果没有则可能是这东西不成立，或者说成立的条件与我们预想的不一致。这往往是十分复杂的）。从而写出

```lean
have eq1 : n + m = m + n := Nat.add_comm n m
```

> **```have``` 关键字**： 它表示应用某个定理的固定格式。后面一般接三个部分
>
> 1. ```:=``` 之后的表示应用的定理。 这里我们需要对 ```n``` 、```m``` 应用交换律 ```Nat.add_comm``` ，即 ```Nat.add_comm n m``` 。
> 2. ```:``` 之后， ```:=``` 之前的部分是定理应用后的结论。
> 在这里我们希望得到的结论是：交换律 ```n + m = m + n``` 成立。这一部分可以省略不写。但建议写出来，这可以帮助你明确思路，以防止自动推断失败，与推断出的并不是你想要的。
> 3. ```have``` 之后， ```:``` 之前的部分是名称。它代指了对这个定理应用的结果。起一个名称方便后续使用。在这个例子中我们起名叫 ```eq1``` 代指对加法交换律的应用结果。这个名称没有特别的含义。

其中标准库中 [```Nat.add_comm```](https://leanprover-community.github.io/mathlib4_docs/Init/Data/Nat/Basic.html#Nat.add_comm) 的声明如下

```lean
theorem Nat.add_comm (n m : Nat) : n + m = m + n
```

可以看出它返回的实例就是前面希望得到的结论 ```n + m = m + n``` 的实例。
在应用交换律之后观察交互区域可以发现，此时的目标并没有改变。而条件区域多出了一个条件 ```eq1 : n + m = m + n``` 。

```lean
1 goal
n m : ℕ
eq1 : n + m = m + n
⊢ n + m = m + n
```

这说明已经成功对 `n`，`m` 应用了定理 `Nat.add_comm` 。
一般将其理解为 `eq1` 证明了命题 `n + m = m + n` 成立。
而整个证明的目标就是证明命题 `n + m = m + n` 成立。
也就是说此时已经很接近结论了， 下面只要说明 ```eq1``` 就是要证的目标就可以了。这一句在 lean 中表达为

```lean
exact eq1
```

> **```exact``` 策略**：它通常用于处理存在性命题。表明已经有一个东西就是我们想要的。它只有一个部分
>
> - ```exact``` 后边的部分，强调表示一些东西存在。
>
> 在这里我们的目标是 ```⊢ n + m = m + n``` ，而恰好 ```eq1``` 就是它。

小结一下，对于 ```proof_1``` 证明首先使用了策略 ```rewrite``` 来展开定义。
接下来使用 ```have``` 关键字来应用标准库中的定理：交换律。
最后使用策略 ```exact``` 来强调说明应用交换律后的结果就是我们要证明的结论。
可以看出 lean 语言的证明过程就是逻辑证明过程的延续。在后面的例子中我们可以更加明确观察到这一点。

##### 定理 ```prof_2```：交换律的另一种证明

下面会给出 `prof_2` 的另一种证明，它与之前的证明 `prof_1` 并没有本质的区别，只是在表述上有区别。

```{lean-playground} :height: 250px
def prop_nat_comm_2 (n m : Nat) : Prop :=
  n + m = m + n

theorem prof_2 (p : Nat) (q : Nat) : prop_nat_comm_2 p q := by
  -- goal prop_nat_comm_2 p q
  rw [prop_nat_comm_2]
  -- goal p + q = q + p
  rw [Nat.add_comm]
```

观察 ```prof_2``` 的声明，

```lean
theorem prof_2 (p : Nat) (q : Nat) : prop_nat_comm_2 p q 
```

容易发现它与 ```prof_1``` 并没有非常大的区别，只是将参数 ```m``` 、 ```n``` 换为了 ```p``` 、 ```q``` 。
实际上这两个参数的名称叫什么并不影响证明，用张宇的话说“叫阿猫阿狗都可以”。
因此它的证明也是类似的

```lean
  -- goal prop_nat_comm_2 p q
  rw [prop_nat_comm_2]
  -- goal p + q = q + p
  rw [Nat.add_comm]
```

第一步依然是使用 ```rewrite``` 展开定义将目标 从 ```prop_nat_comm_2 p q``` 转化为 ```p + q = q + p``` 。而第二步有所不同，直接使用已经证明过的标准库定理 ```Nat.add_comm``` 改写目标。证明就完毕了。实际上这里发生了相当多的过程。

1. ```rewrite``` 会根据你的提示对 ```p``` 、```q``` 使用 ```Nat.add_comm``` ，得到 ```p + q = q + p``` 的实例，这里这个实例并没有像 ```proof_1``` 中被赋予了一个名字 ```eq1``` 。而是一个匿名的存在。
2. 之后  ```rewrite``` 发现 ```p + q = q + p``` 的实例就是目标，从而完成了证明。相当于自动地完成了 ```exact eq1``` 。

从这个例子里可以看出 ```rewrite``` 策略像一个语法糖，它是由更基础的 ```have``` 、 ```exact``` 等，构成的。
这也是策略（tactic）称呼的由来，即包装一些高频操作。
在 lean 中只要跟随常规的证明思路将其翻译成 lean 语言的语法形式就好。不必刻意追求使用策略或回避使用各种策略。

此外需要注意的是 lean 语言的交互过程是不断地转换目标，直到变成某个条件。证明就结束了。类似于

> 1. 为了证明目标1，只要证目标2。 
> 2. 为了证名目标2，只要证目标3.
> 3. ......
> 4. 这个目标就是条件，证毕。

这更加符合人们进行思考与分析证明时的习惯。
而常规的证明过程是将思考倒置，从条件出发逐渐构造结论的过程。
因此经常出现“注意到”这样完全注意不到的过程。
注意到的产生通常是证明书写者逆向分析了目标结论，从而发现只要能够构造出某个结构就可以完成证明。
进而在书写证明的过程中倒置书写导致的结果。
这是分析证明与书写证明的区别。
一般来说分析证明，也就是从结论出发一步步得到结论的过程更容易被人理解。
因此 lean 证明的交互方式选择了不断对目标进行改写，这也有助于雷门理解证明中为什么这里要做这个操作。

#### 证明高斯定理

在开始最终的定理证明之前，我们已经了解了如何进行

1. [如何表达一个命题](#描述问题)
2. [证明书写的基本格式](#证明命题的基本格式)
3. [如何应用其他定理](#如何应用定理)

实际上这已经包含了证明一个定理所有需要用到的能力。
更复杂的定理确实会拥有更加复杂的证明，但也只是命题的表述更加复杂，应用的定理数量更多、条件更多。
其思路本质不会发生变化，都是将数学证明过程翻译为 lean 语言证明的过程。

首先再次回顾原先的证明：

> **命题**： 对任意的自然数 $n$ ，有 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立
>
> **证**：对的自然数 $n$ 使用数学归纳法
>
> 1. 当 $n=0$ 时有 $0=\frac{0\times (0+1)}{2}$ ，成立。
> 2. 假设$n=k$时结论成立，即 $0+1+2+...+k = \frac{k(k+1)}{2}$ 成立。下证明 $n=k+1$ 时有 $0+1+2+...+k+(k+1) = \frac{(k+1)(k+2)}{2}$ 成立。直接计算有
>     $$
>     \begin{aligned}
>           &0+1+2+...+k+(k+1) \\
>          =&(0+1+2+...+k)+(k+1)&加法结合律\\
>          =&\frac{k(k+1)}{2}+(k+1)&使用归纳假设\\
>          =&\frac{k(k+1)+2(k+1)}{2}&通分\\
>          =&\frac{(k+2)(k+1)}{2}&提取公因式\\
>          =&\frac{(k+1)(k+2)}{2}&乘法交换律\\
>     \end{aligned}
>     $$
>     因此 $n=k+1$ 时结论成立。
> 
>     由(1)(2)与归纳法知 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立。$\square$

相应地给出这个证明在 lean 中的翻译。之后再逐句分析

```{lean-playground} :height: 650px
def sum_zero_to_n_formula (n : Nat) : Nat :=
  n * (n + 1) / 2
def sum_zero_to_n_directly (n : Nat) : Nat :=
  if n = 0 then 0 else sum_zero_to_n_directly (n - 1) + n
def prop_eq_sum : Prop :=
  ∀ n : Nat, sum_zero_to_n_directly n = sum_zero_to_n_formula n

theorem proof_sum : prop_eq_sum  := by
  rw [prop_eq_sum]
  intros n
  induction n
  case zero =>
    rw [sum_zero_to_n_directly, sum_zero_to_n_formula]
    simp
  case succ k h =>
    calc
      sum_zero_to_n_directly ( k + 1 ) = sum_zero_to_n_directly k + (k+1) := by
        rw [sum_zero_to_n_directly]
        simp
      _ = k * (k + 1) / 2 + (k+1) := by
        rw [h, sum_zero_to_n_formula]
      _ = (k * (k + 1) + 2 * (k+1)) / 2 := by
        have h1 := Nat.add_mul_div_left (k * (k + 1)) (k+1) (Nat.zero_lt_two)
        rw [<-h1]
      _ = ((k + 2) * (k + 1)) / 2 := by
        have h1 := Nat.add_mul k 2 (k+1) 
        rw [h1]
      _ = (k + 1) * (k + 2) / 2 := by
        rw [Nat.mul_comm]
```

笔者相信大部分人一次性看到这么多的证明都会产生畏难情绪。
实际上这种情绪的产生是再正常不过的。
短时间内看不明白一个东西未免就会产生烦躁心理。
但大部分东西只要一点点看都是能够理解的。
毕竟大部分东西一旦写出来就会变得非常的啰嗦，而其中的核心思想都是非常直观的。

先看证明的固定格式

```lean
theorem proof_sum : prop_eq_sum  := by
```

其含义是书写一个名叫 `proof_sum` 的定理，它证明了命题 `prop_eq_sum` 。
之后书写的都是证明的具体步骤。
此时将光标放在 `by` 之后，可以看到当前证明的目标是 `prop_eq_sum` 。
而为了证明这个命题，就需要看看这个命题代表了什么。因此写出了下面这句 `rewrite`

```lean
-- goal prop_eq_sum
rw [prop_eq_sum] -- goal ∀ (n : Nat), sum_zero_to_n_directly n = sum_zero_to_n_formula n
```

这句 `rewrite` 说按照 `prop_eq_sum` 的定义去替换证明的目标。
因此可以从 lean 的交互窗口观察到目标发生了变化。
而这一部分并没有在自然语言的证明中出现。
这是因为在使用自然语言时，人们会不自觉的完成这部分内容的替换。
比如说当我说“请证明勾股定理这个命题”时，证明者会将“勾股定理这个命题”的直接翻译成“直角三角形两条直角边的平方和等于斜边的平方”。
这个翻译过程一般是不自觉完成的。
而在 lean 中它需要被显示的写出来，也就是这句 `rewrite` 做的事情。

> 注：在 lean 中如果我们声明一个定理（ `theorem` ）名叫 `勾股定理` ，是类似于
>
> ```lean
> theorem 定理勾股定理 : 命题勾股定理 := by 
>   勾股定理的证明过程
> ```
>
> 这样的证明过程。而如果声明一个命题名叫“勾股定理”则是类似于
>
> ```lean
> def 命题勾股定理 : Prop :=
>   直角三角形两条直角边的平方和等于斜边的平方
> ```
> 
> 这样的命题本身。而在自然语言中“勾股定理”这个词汇是根据上下文确定到底表达的是哪一种含义的。
> 换言之在自然语言中，它并不精确的区分这两种含义，这也是我们使用 lean 时会感到不适应的一部分原因。

##### `intros` 策略

下一句 lean 中书写了，

```lean
intros n
```

观察 lean 交互区，可以看到多了一个条件 `n : Nat` ，目标部分少了 `∀ (n : Nat), `。
因此这句话的作用是将目标中的“任意的 $n$ ”从目标转换为条件。这是因为当命题描述了 "对任意的 $n$，" 之后，结论部分就将直接使用 $n$。 因此它可以被看做命题的一部分，也可以看作条件的一部分。而策略 `intros` 负责它们转换。
这种转换通常也是自然语言所欠缺的，毕竟当自然语言描述“对任意的 $n$，”时并不在意它时命题的一部分还是一种先决条件。
而在 lean 中需要精确的区分这些。一般来说，当你想要使用某个东西而它又出现在命题中的时候可以试试这个。

##### `induction` 策略

接下来自然语言的证明说 “对的自然数$n$使用数学归纳法”。 它对应了 lean 中

```lean
induction n
```

策略 `induction` 是一个专门表示归纳法的策略，当遇到需要应用归纳法时就应用这个策略即可。
后边的变量表示对谁应用归纳法，这里是自然数 $n$ 。

在自然语言中应用了归纳法之后，会对初始条件，和递归条件分别进行证明。也就是下面两个目标

> 1. 当 $n=0$ 时，证明结论成立。
> 2. 假设$n=k$时结论成立，下面证明 $n=k+1$ 时结论成立。

而在 lean 中，可以看到对 $n$ 应用了策略 `induction` 之后得到了两个目标 `zero` 和 `succ`。
分别对应的初始条件，和递归条件。

##### `case` 策略

接下来 lean 与自然语言证明一样，都要分一个个目标进行讨论。策略 `case` 就负责这件事情。也就是之后写的

```lean
case zero => sorry
case succ k h => sorry
```

策略 `case` 分为三个部分

1. `case` 之后的条件之前的部分。这部分写明要讨论的目标名称，在这两个 `case` 中分别是 `zero` 和 `succ` 。
2. 名称之后的 `=>` 之前的部分。这部分被称为条件部分。在 `zero` 这个目标中它没有条件，因此就省略不写。
而对于 `succ` 这个目标它有两个条件，分别被称为 `k` 和 `h` 。
3. `=>` 之后的部分，是证明这个目标成立的过程，在这个例子里使用了 `sorry` 来占位。

而在 `case succ` 中出现的条件 `k` 与 `h` ，正是 自然语言中提到的 "假设$n=k$" 和 "假设$n=k$时结论成立" 中的那个结论，即
$0+1+2+...+k = \frac{k(k+1)}{2}$ 这个结论。

##### `case zero` 的证明

首先来看相对简单的归纳法初始条件部分。在自然语言中我们的证明就是将 $n=0$ 带入结论进行验证，就如同下面所描述的

> 当 $n=0$ 时有 $0=\frac{0\times (0+1)}{2}$ ，成立。

那么在 lean 中也是相似的

```lean
-- goal sum_zero_to_n_directly 0 = sum_zero_to_n_formula 0
rw [sum_zero_to_n_directly, sum_zero_to_n_formula] -- goal (if 0 = 0 then 0 else sum_zero_to_n_directly (0 - 1) + 0) = 0 * (0 + 1) / 2
simp
```

第一句就如同自然语言证明一样，首先使用 `rewrite` 将整个命题完全展开。
展开之后可以发现整个目标中不再含有任何变量，比如自然数 $n$ 。
`if` 中的条件 `0 = 0` 可以被判断，等式右边的算式 `0 * (0 + 1) / 2` 也是一个确定的结果。
这时我们希望可以应用某个策略可以化简它们。
lean 中提供了 `simp` 策略来完成这件事情。
它可以化简一些确定的条件判断，或这计算某些不含变量的算式，并判断目标命题是否成立。
在这个例子中它实际上将目标化简为 `0 = 0` 并判断了这个命题成立，从而完成了 `case zero` 的证明。
在完成目标的证明后，可以在交互区域看到 `No goals` 字样。
当看到它的时候就标志着目标证明已经完成。

##### `case succ` 的证明

在这个目标中可以看到证明交互区状态如下：

```lean
1 goal
k : Nat
h : sum_zero_to_n_directly k = sum_zero_to_n_formula k
⊢ sum_zero_to_n_directly (k + 1) = sum_zero_to_n_formula (k + 1)
```

可以看到在目标 `succ` 中已经得到了 $n=k$ 时，结论 `h` 成立。而希望证明的目标是 $n=k+1$ 时，结论成立，也就是目标所示。
在自然语言证明的过程中使用到了拖式计算。那么 lean 中有没有对应的策略呢？ 当然是有的，它就是策略 `calc` 。
而 `calc` 的语法也与拖式计算非常相似，基本格式如下：

```lean
calc
  原始算式 = 变形1 := by 成立的理由1
  _ = 变形2 := by 成立的理由2
  ...
  _ = 变形n := by 成立的理由n
```

其格式基本与拖式计算无异，只是要求每一步都写出可以进行变形的原因。并且使用 `_` 符号进行占位，以免每次都要把 “原始算式” 抄写一遍。

先来看自然语言的部分：

$$
\begin{aligned}
      &0+1+2+...+k+(k+1) \\
     =&(0+1+2+...+k)+(k+1)&加法结合律\\
\end{aligned}
$$

这一步变形利用了加法结合律。目的是出现 $n=k$ 时成立的条件 $0+1+2+...+k$ 为后续使用递归条件做准备。
而在 lean 中也是类似的：

```lean
sum_zero_to_n_directly ( k + 1 ) = sum_zero_to_n_directly k + (k+1) := by
  rw [sum_zero_to_n_directly]
  simp
```

这一步首先使用 `rewrite` 将求和展开 从而出现 $0+1+2+...+k$ 即 `sum_zero_to_n_directly k` 。 
接下来使用 `simp` 策略化简包含 `if` 的部分。

而后自然语言使用归纳假设进行化简

$$
\begin{aligned}
     =&(0+1+2+...+k)+(k+1)\\
     =&\frac{k(k+1)}{2}+(k+1)&使用归纳假设\\
\end{aligned}
$$

对应的 lean 也使用归纳假设 `h` 进行化简，并将化简后得到的 `sum_zero_to_n_formula` 展开为 $\frac{k(k+1)}{2}$ 。

```lean
_ = k * (k + 1) / 2 + (k+1) := by
  rw [h, sum_zero_to_n_formula]
```

在这里证明过程在一步 `rewrite` 中连续进行多次改写。 它与分开写

```lean
rw [h]
rw [sum_zero_to_n_formula]
```

是等价的。改写的顺序是从左到右按照顺序进行的。不熟练的时候可以全都分开写来检查中间的改写过程是否符合预期。

下一步自然语言

$$
\begin{aligned}
     =&\frac{k(k+1)}{2}+(k+1)\\
     =&\frac{k(k+1)+2(k+1)}{2}&通分\\
\end{aligned}
$$

利用通分进行了恒等变换，那么 lean 中也对应的使用通分进行变换。

```lean
_ = (k * (k + 1) + 2 * (k+1)) / 2 := by
  have h1 := Nat.add_mul_div_left (k * (k + 1)) (k+1) (Nat.zero_lt_two)
  rw [<-h1]
```

只是这个变换略显复杂。 
其中 `h1` 省略了类型， 它的类型是 `(k * (k + 1) + 2 * (k + 1)) / 2 = k * (k + 1) / 2 + (k + 1)` 就是我们想要的结论。
而我们应用的定理是 [`Nat.add_mul_div_left`](https://leanprover-community.github.io/mathlib4_docs/Init/Data/Nat/Div/Basic.html#Nat.add_mul_div_left) ，它的声明如下

```lean
theorem Nat.add_mul_div_left(x z : Nat) (H : 0 < y) :
  (x + y * z) / y = x / y + z
```

翻译成自然语言就是

> 定理 `Nat.add_mul_div_left` : 对给定的自然数 $x$ 、 $y$ 、 $z$ ，且命题 $0<y$ 有证明（即命题 $0<y$ 成立），
> 则命题 $\frac{x + y \times z}{y}=\frac{x}{y}+z$ 成立。

而将 $x$ 改写为 $k \times (k + 1)$ ，$y$ 改写为 $2$ ，$z$ 改写为 $k+1$。 就可以得到想要的结论。
而这个定理要了一个非常奇怪的东西 `H : 0 < y`。
根据前面的改写规则，可以知道 $y$ 被改写为 $2$ 。则改写之后为 `H : 0 < 2` 。
因此它是在说它要一个定理，名为 `H` ，内容是“命题 `0 < 2` 成立"。
或者简单理解为它要一个命题 `0 < 2` 的证明。
有了这个证明，才可以应用定理 `Nat.add_mul_div_left` （这是因为当 $y=0$ 时这个命题并不成立）。
相当于这时我们派生了一个子目标，需要证明命题 `0 < 2` 成立。
幸运的是 lean 中已经提供了这个定理。[`Nat.zero_lt_two`](https://leanprover-community.github.io/mathlib4_docs/Init/Data/Nat/Basic.html#Nat.zero_lt_two) ，它的声明如下

```lean
theorem Nat.zero_lt_two :
  0 < 2
```

下面就可以将改写规则与 `Nat.zero_lt_two` 都用于应用定理 `Nat.add_mul_div_left` 得到 `h1`。最后直接反向应用 `h1` 在目标中。就完成了这一步的翻译。这里 `rewrite` 中的符号 `<-` 表示反向应用等式， 它是`rewrite` 语法的一部分。具体是指将目标中所有满足等式右边的项都改写为等式左边的项。有兴趣的读者可以试试删除 `<-` 观察一下目标有什么变化。

> **可推断参数**：可能有细心的读者发现了这里在应用 `Nat.add_mul_div_left` 时并没有传递 `y` 改写成 `2` 这件事。
> 这是因为 `y` 的改写规则可以从其它参数中推断出来。因此不需要显示（重复）的写出。而 `Nat.add_mul_div_left` 的完整声明如下
>
> ```lean
> theorem Nat.add_mul_div_left(x z : Nat) {y : Nat} (H : 0 < y) :
> (x + y * z) / y = x / y + z
> ```
>
> 其中 `{}` 表示的是可推断参数，在常规应用过程中不需要写出。如果想要显示写出可以使用 `@` 应用定理。如
>
> ```lean
> have h1 := @Nat.add_mul_div_left (k * (k + 1)) (k+1) 2 (Nat.zero_lt_two)
> ```
>
> 完整的写出所有的参数有利于理顺思路，并与编译器对答案及时发现认知中存在的问题。

而最后两步的自然语言描述

$$
\begin{aligned}
     =&\frac{k(k+1)+2(k+1)}{2}&\\
     =&\frac{(k+2)(k+1)}{2}&提取公因式\\
     =&\frac{(k+1)(k+2)}{2}&乘法交换律\\
\end{aligned}
$$

都是直接寻找并应用 lean 中对应的定理

```lean
_ = ((k + 2) * (k + 1)) / 2 := by
  have h1 := Nat.add_mul k 2 (k+1)
  rw [h1]
_ = (k + 1) * (k + 2) / 2 := by
  rw [Nat.mul_comm]
```

其中 [`Nat.add_mul`](https://leanprover-community.github.io/mathlib4_docs/Init/Data/Nat/Basic.html#Nat.add_mul) 与 [`Nat.mul_comm`](https://leanprover-community.github.io/mathlib4_docs/Init/Data/Nat/Basic.html#Nat.mul_comm) 的声明如下

```lean
theorem Nat.add_mul(n m k : Nat) :
  (n + m) * k = n * k + m * k
theorem Nat.mul_comm(n m : Nat) :
  n * m = m * n
```

而对应的自然语言翻译留作练习。

此时目标 `succ` 也完成了证明。而自然语言中 “由(1)(2)与归纳法知 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立。” 这一步我们就不需要翻译了。因为在使用 `induction` 策略是已经翻译过了。至此定理 `proof_sum` 完成了命题 `prop_eq_sum` 的证明。

## 总结

回顾写出 lean 中证明的过程，可以发现我们遵循的流程是，

1. 先使用自然语言表述与证明命题。
2. 将自然语言表述中的计算部分与命题部分都翻译为 lean 语言中表达。
3. 将自然语言的证明逐句翻译为 lean 中的证明表达。  

可以看出在这里自然语言依旧占据了举足轻重的作用。
或者说自然语言写不出的证明换成 lean 语言去表达也是表达不出来的。
lean 并非一种万能神教，它只是规范化中另一个有力工具。
而指导这个工具的核心思想依旧是人的思维。
因为 lean 中为了精确性牺牲了很多的易用性，比如命题 `0 < 2` 也是需要证明的。这样的东西积累起来实际上相当打断思路。

### 关于策略 (Tactic)

本文中简略介绍了非常多的策略，如 `rewrite` 、`simp` 、`calc` 、`induction` 等。 
大部分常用策略这里都有提及，一时间无法全部记住是非常正常的。
而在证明的大部分时间里使用的都是 `rewrite` 、 `simp` 、`induction` 这三个。
例如在lean 中 `match` 实际上是 `induction` 的一种语法糖。
其它的策略基本就是当你想用的时候再去寻找。基本上是只要你觉得某个东西非常显然，或者非常麻烦，那么一定有一种策略对应的解决这个问题。

### 关于证明代码

现在我们只是证明了某些命题，这件事情就已经变得非常复杂了。如果想要更进一步证明某段代码的正确性。除了编写代码本身之外，还要再 lean 中额外写出

1. 函数的预期行为或具有的性质，一般称为功能描述。
2. 证明函数具有这样的性质或行为。

这两点都不容易做到。因为它并不是要求使用自然语言描述或者证明，毕竟自然语言的模糊性还是很大的。
它要求使用 lean 去描述与证明。这基本上要求作者处理大量为了精确性所产生的细节。而这带来的工作量还是非常大的。
这还是暂时不谈这些描述与证明的难度，只对工作量的讨论
