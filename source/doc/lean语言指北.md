# 机器定理证明

**机器定理证明（Automated / Interactive Theorem Proving）** 或者说“使用计算机来辅助证明定理”，是指借助计算机来书写、检验乃至自动发现（并证明）数学证明的一类技术。
它的核心思想与“使用计算机来辅助计算”（各种编程语言如C/C++、Rust）相似，都是将自然语言描述的计算/证明过程转化为一种变成语言。
使得我们可以借助计算机的运算能力与准确性来高效准确的完成计算和证明。

按照人与机器的分工，大致分为两类：

- **自动定理证明（ATP, Automated Theorem Proving）**：机器自主搜索证明，典型代表是 SAT/SMT 求解器（如 Z3、CVC5）和一阶逻辑证明器（如 E、Vampire）。它们擅长有限/可判定片段，但难以处理深奥的数学。这不是本文的重点。
- **交互式定理证明（ITP, Interactive Theorem Proving）**：由人给出证明大纲、计算机负责检查每一步并补全细节，代表系统包括 **Lean**、Coq/Rocq、Isabelle/HOL、Agda 等。它能够覆盖任意复杂的现代数学。

而本文主要以[lean](https://leanprover-community.github.io/learn.html)为引子来介绍交互式定理证明，其目标式可以使用lean进行简单的证明书写。

## 使用计算机来辅助计算

在开始使用lean证明各种东西之前，我们先看一个“使用计算机来辅助计算”的例子。 这有助于理解“使用计算机来辅助证明定理”。

### 计算$1+2+3+...+100=?$

如果我们想计算

$$
0+1+2+...+100=?
$$

我们可以在不借助任何计算工具的条件下写出的计算过程

$$
\begin{align}
&0+1+2+3+...+100 \\
=&(0+1)+2+3+...+100 \\
=&(1+2)+3+...+100 \\
=&(3+3)+...+100 \\
=&... \\
=&5050
\end{align}
$$

这个计算过程非常的直观，就是从左到右一个个数加过去。朴实而原始。那么如果我们考虑“使用计算机来辅助计算”，我们就会得到如下程序

```c++
unsigned int sum_one_to_one_hundred() {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= 100; i++) {
        sum += i;
    }
    return sum;
}
```

这段程序也非常的直观，只是将前面人肉计算的过程翻译成了计算机程序罢了。
但这完成了一个非常了不起的过程，我们第一次进行了“使用计算机来辅助计算”，计算机代替我们完成了繁琐的计算过程。

### 计算$0+1+2+...+n=?$

更进一步如果将问题改为

$$1+2+3+...+n=?$$

这时如果我尝试进行人肉计算

$$
\begin{align}
&0+1+2+...+n \\
=&1+2+...+n \\
=&3+...+n \\
=&... \\
\end{align}
$$

会发现我并不知道该加到多少停止，因为这个问题里并没有给出$n$的具体数值。因此只能说

$$请依照加法的计算方法一个个加过去，直到加到了n$$

当然如果我们“使用计算机来辅助计算”，依然可以写出

```c++
unsigned int sum_zero_to_n_directly(unsigned int n) {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= n; i++) {
        sum += i;
    }
    return sum;
}
```

“使用计算机来辅助计算”并不在乎是否给出了一个具体的$n$。因为它是上面人肉计算的一种翻译。这个程序只是将“$请依照加法的计算方法一个个加过去，直到加到了n$”这句话翻译成了计算机理解的形式。

当然经过一些思考还可以写出

```c++
unsigned int sum_zero_to_n_formula(unsigned int n) {
    unsigned int sum = n * (n + 1) / 2;
    return sum;
}
```

这个算法就有些不同了，因为它不再是人肉计算的简单翻译结果。而是使用了更高级的结论简化了计算过程。具体来说是应用了

$$
    0+1+2+...+n = \frac{n(n+1)}{2} 
$$

这个结论非常的基本，以至于我们我们不会怀疑它的正确性。毕竟这已经是书本上学过并验证过的的东西了。

## 使用计算机来辅助证明

那么下面我们就以求和公式为例演示如何“使用计算机来辅助证明”。那么我们是如何进行证明的呢？一种方式是直接使用数学归法。

命题： 对任意的自然数$n$，有 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立

证：

对的自然数$n$使用数学归纳法

1. 当 $n=0$ 时有 $ 0=\frac{0\times (0+1)}{2}$ ，成立。
2. 假设$n=k$时结论成立，即 $0+1+2+...+k = \frac{k(k+1)}{2}$ 成立。下证明 $n=k+1$ 时有 $0+1+2+...+k+(k+1) = \frac{(k+1)(k+2)}{2}$ 成立。直接计算有

    $$
    \begin{aligned}
          &0+1+2+...+k+(k+1) \\
         =&(0+1+2+...+k)+(k+1)&加法结合律\\
         =&\frac{k(k+1)}{2}+(k+1)&使用归纳假设\\
         =&\frac{k(k+1)+2(k+1)}{2}&通分\\
         =&\frac{(k+2)(k+1)}{2}&提取公因式\\
         =&\frac{(k+1)(k+2)}{2}&乘法交换律\\
    \end{aligned}
    $$

    因此 $n=k+1$ 时结论成立。

    由(1)(2)与归纳法知 $0+1+2+...+n = \frac{n(n+1)}{2}$ 成立。$\square$

那么下面我们就把这个证明过程翻译为计算机理解的方式。

### 描述问题

为了方便观察与更加精确的表述问题，像上面使用自然语言去描述问题通常是不够的。因此将命题改写为如下形式。

命题：

$$
    \forall n\in \mathbb{N},\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

> 注：这个过程一般被称为问题的形式化，其目的是在于消除表达的歧义。
当然它也会带来副作用，在训练不足时会觉得这种表述非常别扭且难以理解。
这是正常的，因为形式化表述本身对于熟悉自然语言表述的人来说并不是自然而然的。
不如说任何表述或者规范本身都有其门槛和表达范式。

观察这个命题可以发现它是由命题部分和计算部分组成：

首先拆解第一个命题部分
$$
\forall A, B
$$
其中 $A$ 是 $n \in \mathbb{N}$， $B$ 是 $\sum_{i=1}^n i = \frac{n(n+1)}{2}$。
它描述的是对于任意的 $A$ 条件成立则 $B$ 结论成立。一般我们将其读作对任意的 $A$ 则 $B$ 。

再拆解 $B$ 这个部分

$$
\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$

它由下面三个部分组成，

$$
\begin{aligned}
\sum_{i=1}^n i &&  \frac{n(n+1)}{2} && C = D
\end{aligned}
$$

其中$\sum_{i=1}^n i$、$\frac{n(n+1)}{2}$都是一个计算过程， 而$C=D$是一个命题。

综上，我们得到了命题部分：

$$
\begin{aligned}
 \forall A, B && C = D
\end{aligned}
$$

和计算部分

$$
\begin{aligned}
\sum_{i=1}^n i &&  \frac{n(n+1)}{2} 
\end{aligned}
$$

#### 表述计算部分

下面首先使用lean表达计算部分 $\frac{n (n + 1) }{ 2}$

```{lean-playground}
:height: 200px
def sum_zero_to_n_formula (n : Nat) : Nat :=
    n * (n + 1) / 2

#eval sum_zero_to_n_formula 100
```

其中```sum_zero_to_n_formula```表示了计算部分$\frac{n(n+1)}{2}$。它被定义成一个函数。lean中的一个函数由下面几个部分组成：

1. 关键字```def```表示声明一个函数（计算部分）。这是lean语法的一部分呢
2. ```(n : Nat)``` 表示 ```sum_zero_to_n_formula``` 接受一个类型为```Nat```名称叫做```n```的参数。```Nat```类型在这里表示整数。
3. ```:```与```:=```中间的```Nat```表示函数```sum_zero_to_n_formula```的返回类型为一个整数类型。
4. ```:=```后面表示的是函数的实现。返回 ```n * (n + 1) / 2``` 的计算结果。

我们使用```#eval```这个方法来调用这个函数。在代码右侧的消息（Message）中可以看到它的输出 ```5050``` 代表着从$0$加到$100$的结果是$5050$。

其次来表述另一个计算部分 $\sum_{i=1}^n i$

```{lean-playground}
:height: 200px

def sum_zero_to_n_directly (n : Nat) : Nat :=
  if n = 0 then 
    0 
  else 
    sum_zero_to_n_directly (n - 1) + n

#eval sum_zero_to_n_directly 100
```

可以看到我们表述的是递归定义

$$
f(n) =
\begin{cases}
0, & n = 0 \\
f(n-1) + n, & n = 1,2,3,...
\end{cases}
$$

其中 $f$ 表示了函数 $sum\_zero\_to\_n\_directly$ 。
这是因为在lean中 $\sum_{i=1}^n i$ 并非不可以写成直接表达，只是递归表达相对于直接表达在证明过程中会更加容易。
换言之，lean的表达范式是推荐优先写成递归形式的。
这一点与其它编程语言刚好相反，通常在其它语言中会使用专门的技巧尽可能的消除递归表达。而在lean中请尽量保留递归表达。

#### 表述命题部分

在表述这个复杂命题之前，首先来看一些基础的命题与对应的lean表示

命题:

$$
\begin{aligned}
    1+1&=2\\
    1+1&=3\\
    \forall n\in \mathbb{N}, n &+ 1 = 4\\
    \exists n\in \mathbb{N}, n &+ 1 = 4\\
\end{aligned}
$$

```{lean-playground}
:height: 300px

def prop_one_add_one_eq_two : Prop :=
  1 + 1 = 2

def prop_one_add_one_eq_three : Prop :=
  1 + 1 = 3

def prop_n_add_one_eq_four : Prop :=
  ∀ n : Nat, n + 1 = 4

def prop_exists_n_add_one_eq_four : Prop :=
  ∃ n : Nat , n + 1 = 4

```

对于前两个关于$1+1$的命题，显然我们知道命题： $1+1=2$ 是真命题、命题： $1+1=3$ 是假命题。在lean中```Prop```不会限制一个假命题的表达。即命题无论真假都可以被表述。

对于后两个命题，分别表述了任意与存在。lean是数学表述的直接翻译。

|lean描述|数学描述|
|-|-|
|```∀ n : Nat```|$\forall n\in \mathbb{N}$|
|```∃ n : Nat```|$\exists n\in \mathbb{N}$|

最后我们来表达 $\forall n \in \mathbb{N} \sum_{i=1}^n i = \frac{n(n+1)}{2}$ 中命题部分 $A = B$，其中 $A$，$B$ 是之前表述过的计算部分。

```{lean-playground}
:height: 250px
def sum_zero_to_n_formula (n : Nat) : Nat := 
  n * (n + 1) / 2

def sum_zero_to_n_directly (n : Nat) : Nat := 
  if n = 0 then 0 else sum_zero_to_n_directly (n - 1) + n

def prop_eq_sum : Prop :=
  ∀ n : Nat, sum_zero_to_n_directly n = sum_zero_to_n_formula n

#check prop_eq_sum
```

与计算部分表达类似， 依旧定义了一个函数 ```prop_eq_sum```。
只是与之前的函数不同它的返回类型不再是```Nat```（自然数），而是一个新类型```Prop```(propositions，命题)。因此 ```prop_eq_sum``` 描述的正是命题：

$$
    \forall n\in \mathbb{N},\sum_{i=1}^n i = \frac{n(n+1)}{2}
$$
本身。



命题:
    $$
        1+1=2\\
        1+1=3\\
        \forall n\in \mathbb{N}, n + 1 = 4\\
        \exists n\in \mathbb{N}, n + 1 = 4
    $$






伟大的高斯告诉我们可以把这个序列颠倒着再加一遍。因此我们得到了证明：

$$
\begin{aligned}
&\begin{aligned}
 &0&+&1    &+&2    &+&...&+&(n-2)&+&(n-1)&+&n\\
+&n&+&(n-1)&+&(n-2)&+&...&+&2    &+&1    &+&0\\
\end{aligned}
= n \times (n + 1) \\
\Rightarrow &2\times (1+2+3+...+n) = n \times (n+1)\\
\Rightarrow &1+2+3+...+n = \frac{n(n+1)}{2}  \quad \square
\end{aligned}
$$



## 类型<->命题、实例<->证明

## abc





theorem eq_symm (A : Type) (n m : A) (h : n = m) : m = n := by
    let h0 := @Eq.rec
    let h1 := @h0 A
    let h2 := @h1 n
    let h3 := @h2 (fun (y' : A)=> fun (h : n = y' ) => y' = n )
    let h4 := @h3 (Eq.refl n)
    let h5 := @h4 m
    let h6 := @h5 h
    exact h6
```


机器定理证明的价值主要体现在三方面：一是**绝对的可靠性**——一旦证明通过内核检查，正确性几乎等同于逻辑公理本身的可信度；二是**大规模数学的可维护性**——像 `mathlib` 这样的形式化库已经积累了数十万条定理，可被组合复用；三是**软硬件验证**——CompCert 编译器、seL4 微内核、Intel 处理器算法等关键系统都依赖此类工具给出形式化正确性保证。近年来 Kevin Buzzard 等人发起的「把现代数学搬进 Lean」运动，以及 Terence Tao 用 Lean 形式化自己最新论文的实践，让机器定理证明从计算机科学的小众工具，逐渐成为数学研究本身的一部分。

本指北将以 **Lean 4** 为载体，从「类型即命题、程序即证明」这一 Curry–Howard 视角出发，带你逐步上手机器定理证明。

对于人来说，证明写成这个样子基本就够用了。



换言之，在大部分场合下人们都认为这个证明过程证明了上述命题。
但对于机器来说这里有好多模棱两可的地方，比如

1. $n=1.5$行不行，$n=-1$行么？
2. $0+1+2+...+n = n+(n-1)+(n-2)+...+0$ 怎么就成立了？(小朋友经常问这个问题)

面对这些问题，肯定有人要问这不是抬杠么。$n$当然是个自然数，加法交换律成立，$0+1+2+...+n$当然等于$n+(n-1)+(n-2)+...+0$。 但实际上这些被我们认知为理所当然的东西，是我们经年累月的训练的得到的直觉（毕竟小朋友对交换律的掌握并不熟练，就会觉得并不显然）。
但很不幸的是计算机是一个非常傻的东西。不告诉它$n$是自然数，它会不知道$n$是什么。
不说明 (2) 成立它就是不知道这是连续应用加法交换律的结果。
换言之好多人觉得十分显然的东西，计算机它觉得完全不显然。因此将条件补充完整后得到命题
