机器定理证明¶

机器定理证明（Automated / Interactive Theorem Proving） 或者说“使用计算机来辅助证明定理”，是指借助计算机来书写、检验乃至自动发现（并证明）数学证明的一类技术。 它的核心思想与“使用计算机来辅助计算”（各种编程语言如C/C++、Rust）相似，都是将自然语言描述的计算/证明过程转化为一种变成语言。 使得我们可以借助计算机的运算能力与准确性来高效准确的完成计算和证明。

按照人与机器的分工，大致分为两类：

  • 自动定理证明（ATP, Automated Theorem Proving）：机器自主搜索证明，典型代表是 SAT/SMT 求解器（如 Z3、CVC5）和一阶逻辑证明器（如 E、Vampire）。它们擅长有限/可判定片段，但难以处理深奥的数学。这不是本文的重点。

  • 交互式定理证明（ITP, Interactive Theorem Proving）：由人给出证明大纲、计算机负责检查每一步并补全细节，代表系统包括 Lean、Coq/Rocq、Isabelle/HOL、Agda 等。它能够覆盖任意复杂的现代数学。

而本文主要以lean为引子来介绍交互式定理证明，其目标式可以使用lean进行简单的证明书写。

基础知识与前置技能¶

阅读本文需要掌握一些基本的知识：

  1. 能够阅读C/C++的基本语法

  2. 能够使用自然/数学语言写出一些基本的证明

  3. 可以使用简单的三段式逻辑推理

上述要求应该不难满足。 另外lean在线网站可以在免去复杂的安装过程的同时体验一下lean的能力。是一个非常方便的工具。

使用计算机来辅助计算¶

在开始使用lean证明各种东西之前，我们先看一个“使用计算机来辅助计算”的例子。 这有助于理解“使用计算机来辅助证明定理”。

计算\(1+2+3+...+100=?\)¶

如果我们想计算

\[ 0+1+2+...+100=? \]

我们可以在不借助任何计算工具的条件下写出的计算过程

\[\begin{split} \begin{align} &0+1+2+3+...+100 \\ =&(0+1)+2+3+...+100 \\ =&(1+2)+3+...+100 \\ =&(3+3)+...+100 \\ =&... \\ =&5050 \end{align} \end{split}\]

这个计算过程非常的直观，就是从左到右一个个数加过去。朴实而原始。那么如果我们考虑“使用计算机来辅助计算”，我们就会得到如下程序

unsigned int sum_one_to_one_hundred() {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= 100; i++) {
        sum += i;
    }
    return sum;
}

这段程序也非常的直观，只是将前面人肉计算的过程翻译成了计算机程序罢了。 但这完成了一个非常了不起的过程，我们第一次进行了“使用计算机来辅助计算”，计算机代替我们完成了繁琐的计算过程。

计算\(0+1+2+...+n=?\)¶

更进一步如果将问题改为

\[1+2+3+...+n=?\]

这时如果我尝试进行人肉计算

\[\begin{split} \begin{align} &0+1+2+...+n \\ =&1+2+...+n \\ =&3+...+n \\ =&... \\ \end{align} \end{split}\]

会发现我并不知道该加到多少停止，因为这个问题里并没有给出\(n\)的具体数值。因此只能说

\[请依照加法的计算方法一个个加过去，直到加到了n\]

当然如果我们“使用计算机来辅助计算”，依然可以写出

unsigned int sum_zero_to_n_directly(unsigned int n) {
    unsigned int sum = 0;
    for(unsigned int i = 0; i <= n; i++) {
        sum += i;
    }
    return sum;
}

“使用计算机来辅助计算”并不在乎是否给出了一个具体的\(n\)。因为它是上面人肉计算的一种翻译。这个程序只是将“\(请依照加法的计算方法一个个加过去，直到加到了n\)”这句话翻译成了计算机理解的形式。

当然经过一些思考还可以写出

unsigned int sum_zero_to_n_formula(unsigned int n) {
    unsigned int sum = n * (n + 1) / 2;
    return sum;
}

这个算法就有些不同了，因为它不再是人肉计算的简单翻译结果。而是使用了更高级的结论简化了计算过程。具体来说是应用了

\[ 0+1+2+...+n = \frac{n(n+1)}{2} \]

这个结论非常的基本，以至于我们我们不会怀疑它的正确性。毕竟这已经是书本上学过并验证过的的东西了。

使用计算机来辅助证明¶

那么下面我们就以求和公式为例演示如何“使用计算机来辅助证明”。那么我们是如何进行证明的呢？一种方式是直接使用数学归法。

命题： 对任意的自然数\(n\)，有 \(0+1+2+...+n = \frac{n(n+1)}{2}\) 成立

证：

对的自然数\(n\)使用数学归纳法

  1. 当 \(n=0\) 时有 \( 0=\frac{0\times (0+1)}{2}\) ，成立。

  2. 假设\(n=k\)时结论成立，即 \(0+1+2+...+k = \frac{k(k+1)}{2}\) 成立。下证明 \(n=k+1\) 时有 \(0+1+2+...+k+(k+1) = \frac{(k+1)(k+2)}{2}\) 成立。直接计算有

    \[\begin{split} \begin{aligned} &0+1+2+...+k+(k+1) \\ =&(0+1+2+...+k)+(k+1)&加法结合律\\ =&\frac{k(k+1)}{2}+(k+1)&使用归纳假设\\ =&\frac{k(k+1)+2(k+1)}{2}&通分\\ =&\frac{(k+2)(k+1)}{2}&提取公因式\\ =&\frac{(k+1)(k+2)}{2}&乘法交换律\\ \end{aligned} \end{split}\]

    因此 \(n=k+1\) 时结论成立。

    由(1)(2)与归纳法知 \(0+1+2+...+n = \frac{n(n+1)}{2}\) 成立。\(\square\)

那么下面我们就把这个证明过程翻译为计算机理解的方式。

描述问题¶

为了方便观察与更加精确的表述问题，像上面使用自然语言去描述问题通常是不够的。因此将命题改写为如下形式。

命题：

\[ \forall n\in \mathbb{N},\sum_{i=1}^n i = \frac{n(n+1)}{2} \]

注：这个过程一般被称为问题的形式化，其目的是在于消除表达的歧义。 当然它也会带来副作用，在训练不足时会觉得这种表述非常别扭且难以理解。 这是正常的，因为形式化表述本身对于熟悉自然语言表述的人来说并不是自然而然的。 不如说任何表述或者规范本身都有其门槛和表达范式。

观察这个命题可以发现它是由命题部分和计算部分组成：

首先拆解第一个命题部分

\[ \forall A, B \]

其中 \(A\) 是 \(n \in \mathbb{N}\)， \(B\) 是 \(\sum_{i=1}^n i = \frac{n(n+1)}{2}\)。 它描述的是对于任意的 \(A\) 条件成立则 \(B\) 结论成立。一般我们将其读作对任意的 \(A\) 则 \(B\) 。

再拆解 \(B\) 这个部分

\[ \sum_{i=1}^n i = \frac{n(n+1)}{2} \]

它由下面三个部分组成，

\[ \begin{aligned} \sum_{i=1}^n i && \frac{n(n+1)}{2} && C = D \end{aligned} \]

其中\(\sum_{i=1}^n i\)、\(\frac{n(n+1)}{2}\)都是一个计算过程， 而\(C=D\)是一个命题。

综上，我们得到了命题部分：

\[ \begin{aligned} \forall A, B && C = D \end{aligned} \]

和计算部分

\[ \begin{aligned} \sum_{i=1}^n i && \frac{n(n+1)}{2} \end{aligned} \]

表述计算部分¶

下面首先使用lean表达计算部分 \(\frac{n (n + 1) }{ 2}\)

其中sum_zero_to_n_formula表示了计算部分\(\frac{n(n+1)}{2}\)。它被定义成一个函数。lean中的一个函数由下面几个部分组成：

  1. 关键字def表示声明一个函数（计算部分）。这是lean语法的一部分呢

  2. (n : Nat) 表示 sum_zero_to_n_formula 接受一个类型为Nat名称叫做n的参数。Nat类型在这里表示整数。

  3. :与:=中间的Nat表示函数sum_zero_to_n_formula的返回类型为一个整数类型。

  4. :=后面表示的是函数的实现。返回 n * (n + 1) / 2 的计算结果。这里返回实例并不需要写 return 关键字，这是与 C/C++ 不同的地方。

我们使用#eval这个方法来调用这个函数。在代码右侧的消息（Message）中可以看到它的输出 5050 代表着从\(0\)加到\(100\)的结果是\(5050\)。

其次来表述另一个计算部分 \(\sum_{i=1}^n i\)

可以看到我们表述的是递归定义

\[\begin{split} f(n) = \begin{cases} 0, & n = 0 \\ f(n-1) + n, & n = 1,2,3,... \end{cases} \end{split}\]

其中 \(f\) 表示了函数 \(sum\_zero\_to\_n\_directly\) 。 这是因为在lean中 \(\sum_{i=1}^n i\) 并非不可以写成直接表达，只是递归表达相对于直接表达在证明过程中会更加容易。 换言之，lean的表达范式是推荐优先写成递归形式的。 这一点与其它编程语言刚好相反，通常在其它语言中会使用专门的技巧尽可能的消除递归表达。而在lean中请尽量保留递归表达。

表述命题部分¶

在表述这个复杂命题之前，首先来看一些基础的命题与对应的lean表示

命题:

\[\begin{split} \begin{aligned} 1+1&=2\\ 1+1&=3\\ \forall n\in \mathbb{N}, n &+ 1 = 4\\ \exists n\in \mathbb{N}, n &+ 1 = 4\\ \end{aligned} \end{split}\]

可以看到命题的表述与之前的计算函数类似。更准确的说是有一个函数，它的返回类型为命题。 它与之前的函数最大的不同它的返回类型不再是Nat（自然数），而是一个新类型Prop(propositions，命题)。 而实例 1 + 1 = 2 、1 + 1 = 3 、∀ n : Nat, n + 1 = 4 和 ∃ n : Nat , n + 1 = 4 都是类型Prop 的实例。就如同实例 1 、2 和 3 都是类型 Nat 的实例一样。

实例与类型：为了方便我们使用下面的记法表达实例与类型的关系。

\[\begin{split} > 1 : Nat\\ > 1 + 1 = 3 : Prop\\ > ∃ n : Nat , n + 1 = 4 : Prop > \end{split}\]

其中 : 左边表示实例，右边表示类型。 分别读作

\[\begin{split} > 1 是Nat的实例\\ > 1+1=3 是Prop的实例\\ > 存在n是Nat的实例，使得n+1=4 是Prop 的实例 >\end{split}\]

其中前两个比较简单，第三个则相对复杂，因为它涉及了嵌套关系

\[\begin{split} > n : Nat\\ > ∃ _ , n + 1 = 4 : Prop >\end{split}\]

将 ∃ n : Nat , n + 1 = 4 这个东西视为一个整体才是Prop的实例。 也就是说存在n是Nat的实例，使得n+1=4作为一个整体才是Prop的实例。 单独拿出任何一部分都不是这个式子中期望表达的Prop的实例。

注：实际上 Prop 这个类型并没有什么特殊的意义，它和众多类型如 Nat 没有区别。只是使用lean语言的人觉得这个东西可以被证明，因此单独拿出来讨论。

对于前两个关于\(1+1\)的命题，显然我们知道命题： \(1+1=2\) 是真命题。命题： \(1+1=3\) 是假命题。 在lean中Prop不会限制一个假命题的表达。即命题无论真假都可以被表述。当然假命题是证不出来的。

对于后两个命题，分别表述了任意与存在。lean是数学表述的直接翻译。只是将 \(\in\) 替换为了 \(:\) ，其它部分并没有变化。

lean描述

数学描述

∀ n : Nat

\(\forall n\in \mathbb{N}\)

∃ n : Nat

\(\exists n\in \mathbb{N}\)

注1：严格来说这个表述不够严谨，因为Nat类型并非一个集合，n : Nat 表述的是 \(n\) 是 Nat 的一个实例\(n\)。 n ∈ (setOf fun _ : Nat => true) 表达的才是 \(n\) 属于的由Nat所有元素构成的集合。 这两者是完全等价的，因此本文都选用更易于理解的一种。

接下来再来看另一组返回命题的函数

其中(1)(2)是一组，命题(3)(4)是另一组。其中(1)(2)这一组，描述的是有一个函数接受两个类型为Nat

\[ 如果n,m是自然数，则 n+m=m+n \]

强调的是如果你给我 \(n\) 和 \(m\) 则还给你命题\(n+m=m+n\)， 当然如果你給我 \(p\) 和 \(q\) 则会还给你命题 \(p+q=q+p\)。 至于参数 \(n\) ，\(m\) 从哪里来这个命题并不关心，这一部分由构造命题的人从外部传入。

而命题(3)(4)描述的就是命题：

\[ \forall n,m \in \mathbb{N},n+m=m+n \]

它不依赖任何外部传入的参数，直接了当的说明想表达的命题。

在这里我们使用了 #check 来检查某个实例的类型，在lean中它与 #eval 一样会显示一些消息在message中。其中

  1. #check prop_nat_symm_1 显示了 prop_nat_comm_1 (n m : ℕ) : Prop ，其含义是 prop_nat_comm_1 接受了两个参数 n m : ℕ 后返回一个的类型是Prop的实例。

  2. #check prop_nat_symm_1 1 2 显示了 prop_nat_comm_1 1 2 : Prop ，其含义是 prop_nat_comm_1 接受了两个参数 1 与 2 后返回一个类型是Prop的实例。。实际上我们还知道该实例就是 1+2=2+1 。 当然#check 并不显示这些。

  3. #check prop_nat_symm_4 1 2 显示了 prop_nat_comm_4 : Prop ，其含义是 prop_nat_comm_4 不需要任何输入就返回一个类型是Prop的实例。

最后我们来表达 \(\forall n \in \mathbb{N} \sum_{i=1}^n i = \frac{n(n+1)}{2}\) 中命题部分 \(\forall A, B\) 、 \(C = D\)，其中 \(A\) 、\(B\) 、 \(C\) 、 \(D\) 是之前表述过的部分，汇总如下。

至此我们在lean中完成了命题的命题的表述。一般来说，在lean中，请将一个定义或实现拆解为尽可能小的组成部分。 确保一个定义或实现中只关心一件事情。 这样做是因为lean由于需要保证精确性，许多复杂的表述并不容易在证明处理过程中处理。 这有一点像在刚开始学习证明或计算时，老师可能会要求学生写出每一步所有使用的定理，并禁止省略任何步骤。 熟练之后则无需再注意这些规范，因为它已经成为了你的直觉的一部分。

至此终于使用lean完成了对命题的表述，下面终于可以开始问题的证明了。

证明命题¶

证明的书写总有固定的格式（无论会不会证，先写个证字把条件抄一遍再说）lean也不会例外。首先给出证明的基本样式：

这里出现了一些之前没有的关键字 sorry 、 by 和 theorem 。

theorem 关键字¶

表示这是一个定理，它后面会接着下面几个部分，

  1. 定理名称：在这个例子中定理的名称叫 proof_eq_sum

  2. 定理需要的参数：它的位置在定理名称之后 : 之前，在这个例子中，定理不要额外的参数。 之后我们会看到需要参数的定理。

  3. 定理结论：它是位于 : 于 := 之间的部分。在这个定理就是我们希望证明的命题prop_eq_sum。在现阶段我们希望他是一个 Prop 的实例。

  4. 证明过程：它位于 := 之后。 在这个例子里是 by sorry ， 它由两个部分组成：一部分是 by 有点类似于证明开头的 证： 字。可以暂时理解成无论需要证明什么都需要遵循的固定格式。另一部分 sorry 表示我暂时不知道怎么证明它，写个东西占个位置。（有点类似于我觉得它是对的但地方太小了写不下）

by 关键字¶

可以简单理解成证明的一种固定格式。对于刚开始理解可以不去深究它。

策略模式（Tactic Mod）：by 关键字实际上是策略模式的开关。 添加它表示我们使用策略模式进行证明。在策略模式中我们可以使用很多策略进行证明，如连续使用交换律、结合律的ring，对函数/定理按定义展开的rewrite / rw，等。 更多的策略可以查看策略汇总。这里不再详述。

项式模式（Term Mode）:不添加则停留在项式模式（Term Mode）。它要求使用者手动的构造所有需要的东西。因此需要使用者熟悉所有可能的定理与其条件。这一般来说是一个非常高的要求。因此在实际使用之会使用 let 关键字手动在策略模式下使用它，以减少证明所需的复杂度。

sorry 关键字¶

它的用法就是在函数不知道怎么实现，或者证明不知道怎么证的时候用来占位置的符号。 在本例中函数里使用的 sorry 是为了节约篇章而省略重复的一种手段。theorem 中出现的 sorry 则是暂时不知道如何证明的占位符。

如何应用定理¶

在开始正式证明更复杂的东西之前，首先要学会如何应用已经存在的定理。 毕竟我们没有必要手搓所有的定理，而且这些定理很多并不好证明。稍有不慎就会遇上大量未曾理解或者不曾思考的问题。给我们的学习信心带来大量的打击。为了暂时回避这个事情。我们就要站在巨人肩膀上更进一步。为此首先来看一点简单的命题与证明。

伟大的高斯告诉我们可以把这个序列颠倒着再加一遍。因此我们得到了证明：

\[\begin{split} \begin{aligned} &\begin{aligned} &0&+&1 &+&2 &+&...&+&(n-2)&+&(n-1)&+&n\\ +&n&+&(n-1)&+&(n-2)&+&...&+&2 &+&1 &+&0\\ \end{aligned} = n \times (n + 1) \\ \Rightarrow &2\times (1+2+3+...+n) = n \times (n+1)\\ \Rightarrow &1+2+3+...+n = \frac{n(n+1)}{2} \quad \square \end{aligned} \end{split}\]

类型<->命题、实例<->证明¶

abc¶

theorem eq_symm (A : Type) (n m : A) (h : n = m) : m = n := by let h0 := @Eq.rec let h1 := @h0 A let h2 := @h1 n let h3 := @h2 (fun (y’ : A)=> fun (h : n = y’ ) => y’ = n ) let h4 := @h3 (Eq.refl n) let h5 := @h4 m let h6 := @h5 h exact h6


机器定理证明的价值主要体现在三方面：一是**绝对的可靠性**——一旦证明通过内核检查，正确性几乎等同于逻辑公理本身的可信度；二是**大规模数学的可维护性**——像 `mathlib` 这样的形式化库已经积累了数十万条定理，可被组合复用；三是**软硬件验证**——CompCert 编译器、seL4 微内核、Intel 处理器算法等关键系统都依赖此类工具给出形式化正确性保证。近年来 Kevin Buzzard 等人发起的「把现代数学搬进 Lean」运动，以及 Terence Tao 用 Lean 形式化自己最新论文的实践，让机器定理证明从计算机科学的小众工具，逐渐成为数学研究本身的一部分。

本指北将以 **Lean 4** 为载体，从「类型即命题、程序即证明」这一 Curry–Howard 视角出发，带你逐步上手机器定理证明。

对于人来说，证明写成这个样子基本就够用了。



换言之，在大部分场合下人们都认为这个证明过程证明了上述命题。
但对于机器来说这里有好多模棱两可的地方，比如

1. $n=1.5$行不行，$n=-1$行么？
2. $0+1+2+...+n = n+(n-1)+(n-2)+...+0$ 怎么就成立了？(小朋友经常问这个问题)

面对这些问题，肯定有人要问这不是抬杠么。$n$当然是个自然数，加法交换律成立，$0+1+2+...+n$当然等于$n+(n-1)+(n-2)+...+0$。 但实际上这些被我们认知为理所当然的东西，是我们经年累月的训练的得到的直觉（毕竟小朋友对交换律的掌握并不熟练，就会觉得并不显然）。
但很不幸的是计算机是一个非常傻的东西。不告诉它$n$是自然数，它会不知道$n$是什么。
不说明 (2) 成立它就是不知道这是连续应用加法交换律的结果。
换言之好多人觉得十分显然的东西，计算机它觉得完全不显然。因此将条件补充完整后得到命题



下面就可以正式的将证明翻译成计算机可以理解的形式。
